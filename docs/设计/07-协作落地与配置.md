# 七、协作、落地与配置

## 16. 多人协作（4–5 人）

| 内容 | git | 原因 |
|---|---|---|
| `.flow/{kernel,bin,config.yml,refs,skills}`、`.flow/profile.yml` | 提交 | 全队共用流程、规则与项目规范 |
| `specs/`、`changes/`、`lessons/` | 提交 | 团队真源与共享知识 |
| `.flow/runs/` | gitignore | 多人运行流水进 git 是纯噪声 |

- **命名**：change / lesson 用 `<日期>-<负责人>-<短名>`（如 `20260625-owen-login`）避免撞车。
- **隔离**：run 状态按工作目录哈希隔离，并以 session_id 作软属主锁；一个工作目录同一时刻一个活跃循环。多 change 并行用 git worktree（不同工作目录 → 不同 run）。
- **自测底线**：R0 冒烟；R1 单测；R2 单测+集成；R3 加关键路径 E2E（均取自 profile）。
- **gate**：R0/R1 不拦；R2 在"设计完成"设一可选确认点；R3 双确认点。〔gate 在 MVP 为约定，由编排者在 `loop-start` 前人工确认；机制化拦截为路线图。〕

## 17. 目录结构

```
项目根/
├─ specs/                      当前真源(openspec)，R2/R3 才有        [AI · 提交]
├─ changes/<id>/               delta规格/design/tasks/research        [AI · 提交]
├─ lessons/                    可复用经验(L3)                        [AI · 提交]
├─ docs/                       人类交付物 + Mermaid                  [人 · 提交]
└─ .flow/
   ├─ config.yml              通用契约配置(扁平点号键)               [提交]
   ├─ profile.yml             项目自适应规范(L2，人审)              [提交]
   ├─ kernel/
   │  ├─ lib.sh                共享库                                [提交]
   │  ├─ router.sh             UserPromptSubmit: 判级 + 召回          [提交]
   │  ├─ loop-controller.sh    Stop: Oracle 闭环引擎(L1)             [提交]
   │  ├─ checkpoint.sh         PreCompact: 落盘 state                [提交]
   │  ├─ restore.sh            SessionStart: 恢复 state              [提交]
   │  └─ selftest.sh           内核自测                              [提交]
   ├─ bin/flow                 CLI(tier/loop-*/profile-check)        [提交]
   ├─ refs/antipatterns.md     反模式目录(按 profile 语言包加载)      [提交]
   ├─ skills/                  各 SKILL.md + _contract.md            [提交]
   └─ runs/<cwd哈希>/          loop.yml/state.json/loop-history      [本地 · gitignore]
```

## 18. 安全与护栏

Flow 的 Oracle 闭环参考了 Stop-hook 自治循环的思路，并在以下几点上做了更克制的工程化：

| 护栏 | 机制 |
|---|---|
| 默认有界 | 每 tier 设迭代上限；到上限负责任移交，不无限循环 |
| Oracle 来源受控 | 验证命令只来自人审过的 profile，agent 不在循环中自造 |
| 无 Oracle 不进循环 | 装配不出验证命令的 tier 退回人类确认，不退化为信任自报 |
| 仅主会话生效 | `SubagentStop` 与含 `parent_session_id` 的事件早退，子代理结束不触发循环 |
| 孤儿回收 | 循环态超过 `loop.stale_seconds` 视为残留，回收放行 |
| 会话软锁 | 循环只对属主会话生效，非属主会话不干预 |
| 收敛停滞检测 | 同一批文件连续多轮被改 → 强制根因步 |

**已知限制**：①验证命令超时仅终止直接子进程，长测试运行器可能遗留孤儿进程（进程组终止为路线图）；②机制依赖 Claude Code transcript 的 JSONL 结构，结构变化时信号检测会退化为"无信号"路径（到上限移交兜底）；③一个工作目录一个 run，同目录并行多任务需用 worktree 隔离。

## 19. 实现状态与分期落地

| 阶段 | 内容 | 状态 |
|---|---|---|
| **MVP** | Kernel（router 判级+黏滞+召回 / loop-controller Oracle 闭环+有界+升维 / checkpoint+restore）、`bin/flow`、`_contract`（四权分立）、profile 装配 Oracle + `profile-check`、antipatterns、9 个 skill、三类产物分离、内核自测 | **已实现并通过自测** |
| **P2** | 自动 harvest 触发与去重晋升流水线、召回时间老化、`manifest.json` 汇总与上下文足迹度量、profile 漂移检测、外置 haiku 判级、最小 benchmark | 路线图 |
| **P3** | R3 多 change 编排、worktree 并行、requirement 级冲突检测、integrity-guard 主动改动拦截、lessons 自动喂养 antipatterns/profile | 路线图 |

> 验收核心判据：在真实任务上 Oracle 闭环能自动收敛或有界移交，且主上下文足迹达标（重活在子代理）。内核自测覆盖：非循环放行、无信号阻止+升维+迭代递增、Oracle 通过放行、Oracle 失败阻止+计数、到上限移交、abort/pause、SubagentStop 守护、router 的 rubric/forced/sticky/skip 路径。

## 21. 快速启用

一条命令装进目标项目，装完即用：

```bash
bash install.sh /path/to/your-project
```

安装器自动完成全部配置：①复制 `.flow/`；②把 4 个 hook **幂等合并**进目标的 `.claude/settings.json`（保留既有配置）；③把 `.flow/runs/` 加入 `.gitignore`；④`flow profile-init` **自动探测**构建/测试命令生成 `profile.yml`（识别 npm/pnpm/yarn/bun、Cargo、Go、Python、Makefile）。

之后**正常对话即可，无需记任何命令**：router 每轮自动判级并注入裁决；R2/R3 设计确认后，编排者按注入指引自动驱动 Oracle 闭环。可选手动操作：

```bash
.flow/bin/flow profile-init    # 换了构建工具后重新探测
.flow/bin/flow profile-check   # 自检每 tier 能否装配出 Oracle
.flow/bin/flow loop-status     # 查看当前循环与历史
bash .flow/kernel/selftest.sh  # 验证内核
bash uninstall.sh /path/to/your-project   # 卸载（保留其它配置）
```
