# 四、技能与质量契约

## 8. Skill 体系

每个 skill 是一个 `SKILL.md`，主文件短，深档按需引用。除 router 注入的裁决与 lessons 外全部懒加载。编写纪律：**`description` 只写"何时用"、不写工作流摘要**（否则 agent 会照摘要抄近路跳过正文）；深档按文件名引用、不用 `@`（避免未用先吃上下文）；词数预算（常注入项 < 200 词，其他 < 500 词）。

| Skill | 触发 | 职责 | 执行位置 |
|---|---|---|---|
| `profiler` | profile 缺失/陈旧 | 扫码库 + CI 产/更新项目画像（§10） | 子代理 |
| `research` | R2/R3 按需 | fan-out 检索、抓源、对抗核验、带引用综述 | 子代理，回传结论 |
| `brainstorm` | R2/R3 | 探问题、列候选、辨权衡（不写计划、不定路径） | 主线对话，检索 fan-out 派子代理 |
| `plan` | R2/R3 | 选定方向落成 design.md + tasks.md + 图 | 子代理草拟，人审产物 |
| `implement` | 全部 | builder 开发 + verifier 对抗（角色独立） | 子代理 |
| `self-test` | 全部 | 按 tier 分级，作为 Oracle 验证命令来源 | 子代理 |
| `document` | R1+ | 只产人类交付物，禁流水账 | 主线 |
| `diagram` | R2+ | 仅 Mermaid，内置美观规范（§14） | 主线 |
| `harvest` | 命中信号 | 从历史蒸馏候选经验（§11） | 子代理 |

`brainstorm` 与 `plan` 刻意分开：前者只产方向共识，后者只落可执行设计、不再发散。

## 9. 质量契约 `_contract.md`

一页文档，所有 skill 继承，只含工程结构行为、无任何情绪话术。其多数条款由 §7 Oracle 机制强制，而非靠自律。

- **诚实条款**：禁删/弱化测试或改断言以"变绿"；不确定显式标注；失败如实上报；**完成必带证据**——`<promise>` 只是申请，冻结的 verify 命令才是裁决。
- **四权分立**：builder ≠ verifier；实现者不得改测试/断言/CI/profile；裁决权属独立 Oracle；涉及判据的变更须停下说明。
- **对抗原则**：实现必经独立 verifier 证伪；verifier 默认怀疑，专攻边界/并发/错误路径/隐藏假设；对照 `antipatterns.md`（按 profile 语言包加载）扫描；发现真 bug 提议候选经验。
- **统一升维**：见 §7.3，升维不在第 1 次触发。
- **主动与穷尽**：R0/R1 先做后问；R2/R3 在 gate 处先问后做；修一个 bug 顺手查同类。
- **产物纪律**：三类受众物理隔离（§12），禁把执行流水当文档交付给人。
