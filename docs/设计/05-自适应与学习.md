# 五、自适应与学习

## 10. L2 项目自适应规范：`profile.yml`

Flow 不强加通用规范，而把通用契约特化到本项目。profile 是规范自适应的载体，也是 L1 Oracle 的受控来源。

### 10.1 自举

安装时 `flow profile-init`（亦可随时手动重跑）**自动探测**：从包/构建清单（package.json scripts、Cargo.toml、go.mod、pyproject、Makefile）识别 build/test/lint 命令与包管理器，从顶层目录推断模块边界，从依赖识别语言与反模式包。探测到 build 或 unit 即置 `profile.ready: true`，装完即用；否则置 `false` 并提示手填。需要更高保真度时，可由 `profiler` 子代理进一步读 CI 配置（CI 真正跑的就是验证函数）并人审校正——Oracle 正确性是 L1 全程承重点，错的验证命令会毒化整个循环。`flow profile-check` 自检每 tier 能否装配出 Oracle。

### 10.2 schema（扁平点号键，无 YAML 依赖）

```yaml
profile.ready: true                 # 人审通过后置 true，否则 loop-start 拒绝
languages: typescript,go
module_boundaries: apps/web,services/auth   # 供 rubric 影响面粒度（逗号分隔）
commands.build: pnpm build          # 本项目存在的填命令，不存在留空
commands.unit:  pnpm test:unit
commands.integ: pnpm test:integration
oracle.R0: build smoke              # 每 tier 由哪些命令键拼装（空格分隔）
oracle.R2: build unit integ
conventions.doc_lang: zh-CN
antipattern_packs: typescript,react # 选用语言/框架反模式包
field_meta.commands.integ: 0.6|2026-06-20   # 置信度|最后验证日期（老化用）
```

### 10.3 profile 驱动什么

L1 Oracle（`oracle.<tier>`×`commands.*` 装配 verify）；rubric 影响面粒度（`module_boundaries`）；反模式语言包（`antipattern_packs`）；自测分级；交付物语言（`conventions.doc_lang`）；代码风格约定。

### 10.4 降级与演化

**Oracle 来源降级链**：有 CI → 用 CI 命令；无 CI → 用清单推断；无测试 → Oracle 退化为 build+lint；无可用 verify → 该 tier 退回人类确认（`oracle.*` 留空即可）。**演化**：包清单/CI 变更触发字段校正；`field_meta` 带置信度 + 日期供老化降权。〔自动漂移检测为路线图，当前为按需重跑 profiler。〕

## 11. L3 框架回路：学习闭环

与文件模型同构的三态生命周期，对象是知识：`候选(本地 runs) → 晋升(committed lessons/) → 固化(antipatterns/rubric/profile)`。

- **捕获**：run 末尾，命中信号才蒸馏候选——失败升维 ≥ 3、循环到上限移交、发生返工、人工覆盖 tier、verifier 发现真 bug（阈值见 `config.yml` 的 `harvest.*`）。无信号不产经验，避免灌水。候选含症状/根因/修复/可泛化结论/关联/最后验证日期。
- **晋升**：候选经去重 + 轻量人审入 `lessons/`（committed，团队共享）。
- **召回与固化**：router 按路径/标签 top-k 注入；按时间老化降权（老化非硬删）；复发者固化进 `antipatterns.md` / rubric 阈值 / `profile.yml`，使反模式目录从纯手工变为部分自动喂养。
- **从覆盖学路由**：人工覆盖是 rubric 的带标签信号，周期性回看校准阈值。

> 当前实现：捕获信号、阈值、召回（关键词命中）与 `lessons/` 目录已就位；自动 harvest 触发、去重晋升流水线、召回的时间老化为〔路线图〕，MVP 下由 `harvest` skill 半自动执行。
