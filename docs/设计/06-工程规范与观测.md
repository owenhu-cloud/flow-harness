# 六、工程规范与观测

## 12. 三类产物分离

| 文件 | 受众 | 形态约束 |
|---|---|---|
| `specs/`、`changes/<id>/*`、`lessons/`、`profile.yml` | **AI** | 密集、结构化、可解析；task 带 checkbox、路径明确；无叙事日志 |
| `docs/*` + Mermaid + PR body + CHANGELOG | **人** | 只含结论、设计取舍、图；禁 agent 思考过程/执行流水账 |
| `.flow/runs/<hash>/*` | **审计** | 机器写机器读：tier、迭代、Oracle 结果、状态 |

**明令禁止**：把 agent 执行流水当"文档"交付给人——那是审计数据，不是人类交付物。

---

## 13. OpenSpec 文件模型集成

借用 `specs/`（当前真源）+ `changes/<id>/`（delta 规格，只写差异）+ `propose → apply → archive` 生命周期。**仅 R2/R3 触发**，R0/R1 零仪式。校验/归档等纯机械动作可用 openspec CLI 在上下文外执行。多人改 `specs/` 时，对重叠 requirement 的 delta 应在合并阶段显式暴露冲突、不静默合并〔requirement 级冲突检测为路线图〕。角色：OpenSpec = 状态骨架；Flow = 编排 + 质量 + 学习 + 自适应。

---

## 14. 画图规范（仅 Mermaid）

类型映射（选错类型是最大丑因）：架构→`flowchart`（分层 subgraph）；流程→`sequenceDiagram`；状态机→`stateDiagram`；数据→`erDiagram`。可读约束：单图节点 ≤ ~12（超则拆图）；统一方向；关键路径 `classDef` 高亮；连线必带标签；固定一套四色 `classDef`（主/次/外部/告警）全项目复用。产物落 `docs/`，仅供人看。

> 工程提示：CJK 仅放进节点/边的**标签**（`["…"]`、`-->|…|`），不要用作节点 ID；状态机若遇渲染器兼容问题，可降级为 `flowchart`（本设计第三章升维图即如此）。

---

## 15. 可观测性与质量基线

Flow 不做 token 成本账本（与质量无关），但度量"上下文足迹"——因为"主上下文干净"是核心承诺，不度量就无法验收。基线指标（先设后调）：主上下文足迹（各重活阶段是否子代理执行；核心验收）、L1 循环健康（平均迭代数 / 到上限移交率 / Oracle 一次过率）、返工率（单任务被打回 ≤ 1 次）、隐藏 bug 检出、路由命中率（人工覆盖率 < 20%）、自测覆盖达标率、经验产出/晋升、profile 鲜度。

> 当前 `loop-history.jsonl` 已记录循环健康数据；统一的 `manifest.json` 汇总与上下文足迹度量为〔路线图〕，验收时以"各阶段是否子代理执行"的布尔记录与循环历史为准。
