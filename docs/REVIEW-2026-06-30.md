## 1. 执行摘要

flow-harness **不是花架子**：当项目写入 `docs/flow/verify-cmd` 后，`Stop` hook 的独立 Oracle（`exit 2` 拦截 + 扫 git diff 防删测试）是真实、可审计的机器级门控，这一点比 Claude Code 原生与 spec-driven 类竞品都更具体。作者的工程诚实度也高——几乎所有绕过路径都已在 `flow-oracle.sh:28-69` 的威胁模型里自认。

但必须清醒认识其边界：

> **真正「机器强制」的只有一样：`verify-cmd` 在场时的 Stop Oracle `exit 2`。其余约 80% 的纪律（R0–R3 判档、brainstorm/plan「硬门」、builder≠verifier、跨模型对抗、引用校验、档位地板）全是 SKILL.md 提示词，agent 可以无视。**

它治理的是「压力下偷懒的诚实 agent」，不是「决心作弊的对手」；它把作弊**成本抬高、痕迹留下**，但未密封。代价是几百行 shell + 19 支技能的复杂度，与原生能力大量重叠，并构成新的维护债与攻击面。

**关键数字**：机器强制面 = 1 个门（且 opt-in）；提示词纪律面 = 全部 19 支技能 + 4 维判档 + 5 条质量红线。

## 2. 发现清单（按严重度）

### Critical

#### C-1 🔴 Oracle 无法自保，燃料/基线提交后可被改低
- **证据**：`flow-oracle.sh:240`（A0 仅拦**未提交**的 `verify-cmd`/`robustness-cmd`/`coverage-cmd`/`coverage-min`）；`flow-oracle.sh:38-39` 自认「agent 直接覆写 flow-oracle.sh / verify-cmd / test-count 自身——脚本无法防自身字节被改」；`lessons/oracle-gate-grep-ceiling.md`。
- **根因**：把「可执行 shell 命令 + 可写基线」交给被监管者同仓管理，监管者无法监管自身字节。一旦基线/燃料被提交，改低它只触发「人审 diff」这一纪律级兜底，无机器拦截。
- **影响**：决心作弊者可在一次提交内合法化弱化。
- **缓解现状**：仅靠「建议纳入版本控制 + 人审」（`flow-oracle.sh:69`）。

#### C-2 ⚪ A2 断言计数存在真实假阴性（已实测复现）
- **证据**：`flow-oracle.sh:263-270`。删除侧用 `grep -E '^-[^-]'`（正确排除 `---` 文件头），新增侧用 `grep -E '^\+'`，**未对称排除 `+++` 文件头**。当被改测试文件路径含 `assert`/`expect`/`should` 等关键词时，`+++ b/...` 头行被误计为「新增断言」，抵消真实删除，使 `ASS_DEL > ASS_ADD` 不成立，完整性门 A2 放行。
- **复现**（本轮新鲜输出，见附录 A）：删除 `expect(value).toBe(42)`、文件名 `assert_helpers_test.js` → `ASS_DEL=1 ASS_ADD=1` → 违规**未触发**。
- **判定**：明确的代码不对称 bug，作者意图显然是对称排除（应为 `^\+[^+]`）。Codex 独立指出同一处。
- **修复**：单行——新增侧改 `grep -E '^\+[^+]'`，并补一条针对「文件名含断言关键词」的回归测试。

#### C-3 🔴 TOFU 基线可被永久锁低 + 解析器格式天花板
- **证据**：`flow-oracle.sh:169` 首次绿直接写 `test-count` 建基线（trust-on-first-use）；注释⑥（`:45-46`）自认「先弱化再首绿把地板锁低位，`0 passed` 锁 0 即永久失效」。`extract_count`（`:111-119`）只认 `passed/passing`、`examples`、`ran N tests`、`--- PASS:` —— **JUnit `Tests run:`、Maven Surefire、TAP、多数 JSON reporter 解析不到**；首次无基线即放行（`check_count_baseline` 仅在「已有基线却变不可解析」时打回，`:139-149`）。`extract_coverage`（`:179-189`）同理只认 `total:`/`TOTAL`/`coverage:` 三种格式。
- **影响**：B2/B4 在大量真实 runner/语言栈上静默退化为「无门」。

### Important

#### I-1 🔴 verify-cache 指纹盲区一次穿透全部门
- **证据**：`flow-oracle.sh:201-216` 指纹**刻意排除 `docs/flow/*`**，故开缓存后只改低 `test-count`/`robustness-count`/`coverage-min` 不会使指纹失效 → `:226 exit 0` 跳过 A/B/B2/B3/B4 全部门（注释⑨ `:52-53` 自认）。`--exclude-standard` 还使 `.gitignore` 下的测试依赖（fixtures/.env/本地 DB）改动不触发失效（注释①）。
- **判定**：false-skip 风险与降本收益不成比例。建议弃用，或退回更窄的「git 脏检查」。

#### I-2 ⚪🟡 跨模型派发脆弱：健康检查验不到 flag 漂移 + ARG_MAX
- **证据**：`external-agent.sh:61-72` 健康检查只验「binary 存在 + auth.json 存在」，不验 CLI flag 兼容性——**本次评审亲历**：grok 健康检查会过，但 `grok-build` 拒 `reasoningEffort` 报 HTTP 400（正是 `lessons/stub-argv-test-cant-catch-real-cli-flag-rejection.md` 记录的坑，但健康检查仍未堵）。`:93/:110/:155` 用 `"$(cat "$pf")"` 把整个 prompt 当单个 argv 传，长 diff 撞 ARG_MAX。
- **修复**：派发前跑一次 `--help`/dry-run 探测 flag；prompt 改 stdin/file 传入。

#### I-3 🔴 「硬门」与角色分离是提示词，非机器强制
- **证据**：`flow/SKILL.md:53` 标 brainstorm/plan 为「**硬门**」，但仅靠文字 + 对话确认，无代码阻止 agent 直接 `Skill` 加载 implement；`implement/SKILL.md:56` 自认「hook 无法内省 verifier 是否真跑」，builder≠verifier 完全靠自觉，Oracle 不检查 diff 由谁所改。
- **影响**：流程纪律在「不配合的 agent」面前不成立；属预期设计，但文档对外宣称的「门」语义名实不符（见 §3）。

### Minor

#### M-1 🟡 假阳性卡正常开发
- A2「断言净减少」误伤合法重构（断言抽进 helper、跨文件搬移），需提交豁免文件绕过（`flow-oracle.sh:268-289`）。
- 敏感词地板为**子串匹配**：`tokenizer→token`、`released→release` 误触发地板提示（`flow-reinject.sh:19`，注释自认低害、nudge-only）。

#### M-2 ⚪ 供应链/执行边缘
- Stop hook 无条件 `sh -c "$CMD"` 执行仓库文件中的命令（`flow-oracle.sh:293`），无 allowlist、无确认。装有本插件时克隆**不可信仓库**，其自带的 `docs/flow/verify-cmd` 会在首次收尾时被执行。属「Stop hook 跑项目命令」的固有风险，但因 `verify-cmd` 自动写入、零确认而默认敞开。

#### M-3 ⚪ `stop_hook_active` 用裸子串匹配
- `flow-oracle.sh:77-78` 以 `*'"stop_hook_active":true'*` 子串判定，而非 JSON 解析；脆弱（虽当前输入由 Claude Code 控制，风险低）。

## 4. 过度设计 / 低 ROI

- 🔴 **4D rubric + 19 支技能 + 档位地板**与 Claude Code 原生 **plan mode + Task subagents + Stop hook + TodoWrite** 严重重叠，再包一层仪式。
- 🟡 **Oracle 五门内部复杂度**（bignum 特判、占用数计数、指纹缓存、override 刷新）维护成本高。`lessons/` 整个目录是 dogfood 撞出来的绕过补丁史，既证明作者诚实，也提示「门」未密封。Grok 评语：多数项目 `npm test && git diff --stat | grep -E 'test|spec'` 即可挡 70% 问题。
- ⚪ `verify-cache` + 复杂指纹 + 满屏 threat 注释，增加理解负担（见 I-1，建议弃用）。
- ⚪ `external-agent.sh` 700+ 行服务于默认关闭、极少数 R3 场景。

## 6. 与竞品对比

| 维度 | flow-harness | Claude Code 原生 | spec-kit / BMAD 类 |
|---|---|---|---|
| 完成裁决 | ✅ **独立进程 Oracle + git diff 防作弊**（真差异化） | Stop hook 能做但需自写 | ❌ 多为静态模板/checklist，无运行时门 |
| 抗上下文衰减 | ✅ 每轮 reinject（轻量有效） | TodoWrite/plan mode 部分覆盖 | ❌ 无 |
| 流程结构 | R0–R3 路由 + 画像 + lessons 沉淀 | plan mode + subagents（更轻直接） | ✅ 需求→验收矩阵更系统 |
| 需求追踪/验收 | ⚠️ 弱，多靠文本 | 弱 | ✅ **强**（flow 短板） |
| 机器强制面 | ⚠️ 窄（仅 Oracle） | 窄 | 窄 |

**独特价值**：把反 reward-hacking 从 LLM 自评落成「独立进程 + exit code + git diff 机器裁决」，并加每轮抗衰减 nudge。
**核心短板**：① 提示词纪律面积 ≫ 机器强制面积；② 实现复杂度本身成为新攻击面与维护债；③ 对「同模型 sycophancy + 长上下文衰减」的抵抗仍主要靠文字 nudge。

## 7. 总评

Stop Oracle（`verify-cmd` 在场时）是真东西，作者工程诚实度高。但其定位应被准确理解为**「抬高作弊成本、留下审计痕迹」的本地质量闸**，而非密封的机器强制框架。真正根治 reward-hacking 终究需要**带外 CI（agent 改不到的环境）+ 变异测试 + 人审基线变更**——flow 在「让该多重的流程结构化、让偷懒留痕」这个定位上合格；超出此定位的「机器强制」宣称大半名实不符，应在文档与优先级上据实修正。

## 附录 A：C-2 复现证据（本轮新鲜输出）

输入 diff（删除一处真实断言，文件名含 `assert`）：

```diff
--- a/tests/assert_helpers_test.js
+++ b/tests/assert_helpers_test.js
@@ -1,3 +1,2 @@
 test('x', () => {
-  expect(value).toBe(42)
 })
```

套用 `flow-oracle.sh:269-270` 的计数逻辑：

```
ASS_DEL=1  ASS_ADD=1
A2 triggers violation (DEL>ADD)? NO -> deletion slipped past gate A2
what + side wrongly matched: assert      # 来自 +++ 文件头，而非真实断言
```

→ 删除的 `expect(value).toBe(42)` 被 `+++ b/tests/assert_helpers_test.js` 头行的 `assert` 计数抵消，完整性门 A2 未能拦截。

**诚实保留的边界（非搪塞，Codex 确认无廉价闭合）**：`verify-cache` 开启时，B2（runner 输出通过数）在缓存命中时本就不跑——缓存即"跳过 verify-cmd 执行、无输出可验"；伪造 `.last-green` 可跳过 B 段。这是缓存固有取舍（要根治需重跑 verify-cmd 或仓库外不可伪造根）。已把 **A 完整性门 + A0c 基线门移到缓存命中之前 always 跑**（删/弱化测试、改基线文件即便伪造缓存仍被拦），并明确 `verify-cache` 为**降本优化、非安全边界**（高保真场景关掉）。

## 附录 B：方法学说明

本报告由三个异构模型独立产出后交叉合并：单方提出的发现均经本评审回到源码或实测核实（标 ⚪）；两方/三方独立命中同一处标 🟡/🔴 以示置信。Grok 因 headless sidecar 故障改用内联源码方式取得产出，该故障本身复现了 I-2。所有 file:line 引用基于 `v1.4.0` 当前源码。
