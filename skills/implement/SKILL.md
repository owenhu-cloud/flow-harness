# implement — TDD 构建 + 对抗证伪

**适用档位：R1–R3。** 在 Flow 流程 `brainstorm → plan → implement → verify → document` 中处于 **implement** 位：把**一个 change**（一条 task / 一个 bug）从红测试做到绿、再让独立 verifier 来搞坏它。R2/R3 应已有 `plan` 产出的 `tasks.md`；R1 可直接进来。多 change 的编排不在这——那是 `subagent-driven-development`；本技能只钉**单个 change 的 TDD 节奏与对抗**。

动手前先读 `docs/flow/project.md`（项目画像）：用其中固化的命令、风格约定与**项目特异反模式**，别套通用规范。无画像 → 先用 `profile` 固化（治一刀切）。

两个**角色独立**的子代理，禁止合并（红线 §2）。用 `Task` 工具把 builder 与 verifier 分派给不同子代理。

## 铁律（iron-laws，不可违反）

1. **没有失败的测试，就不写一行生产代码。** 先 RED（写出会失败的测试并亲眼看它失败、确认失败原因正确），再 GREEN（写最少代码让它通过），再 REFACTOR（测试仍绿下清理）。顺序不可颠倒、不可跳。
2. **测试先于实现存在。** 任何「先写实现，事后补测试」一律视为作弊——后补测试立即变绿，证明不了任何东西（它测的是你写出来的东西，不是需求要求的东西）。
3. **旧实现不是参考，是污染源。** 重写/迁移时，旧代码不留作参考、不在写测试时瞄它、不照着改编——那等于后补测试。删掉，从测试重新实现。
4. **builder 不碰测试/断言/CI 去变绿。** 改测试让它过属于 verifier 域；builder 改 = 越权 + 作弊（红线 §1/§2）。
5. **完成由 verify 的新鲜输出裁决，不由「我觉得过了」。** 见下「完成判定」。

TDD 循环的完整纪律（每一步该看到什么、何时算真的 RED）见 `references/tdd-loop.md`。

## builder（构建）

- **动手前先列证伪假设（前置 falsification）**：写实现前先列 2–3 条「这个改动会怎么错」的具体假设（边界 / 并发 / 错误路径 / 用户误用 / 掏空），每条**指认一个可执行 check**（一条测试或命令），先让它红——把「想得更深」绑到可执行反馈而非空想（无外部信号的"自我反思"不可靠）。这些 check 就是本轮 RED 测试的起点。
- 按 `tasks.md` 逐项实现，严格走 RED-GREEN-REFACTOR（铁律 §1）。一次只推进一个最小可测行为。
- 遵项目既有命名/风格——以 `project.md` 的「风格约定」为准，再看周围代码，不自创。
- 每轮动手前 `git diff` + 看 `TodoWrite` 进度，避免重走死路。连续失败按 `flow` 的「升维」表换思路，别原地调参。

## verifier（对抗，独立子代理）

- 默认怀疑，目标是**搞坏它**：边界、并发、错误路径、空值/超大输入/特殊字符/权限不足。
- 对照 `references/antipatterns.md`（按项目语言取分区）**+ `project.md` 的项目特异反模式**双重扫描。
- **不轻信测试存在=测试有效**：抽查测试是否真能 fail——把实现改坏一处看测试是否变红（mutation 心智）。全绿但改坏不报警的测试 = 假测试，按 bug 处理。项目画像若记了 `mutation-cmd`（变异工具），按需跑它做这步而非纯手工抽查——变异不接 Oracle Stop 门（太慢），但它是"①同数语义掏空/假断言"的真根治，覆盖率门(B4)只是分钟内可跑的代理。
- 发现真 bug → 打回 builder；并记成一条候选经验交给 `harvest`。
- **错误路径要落成受机器门守护的测试，不止文本自述**：把 plan `Robustness-Cases` 列的 MUST 异常场景逐条写成测试，并纳入 `docs/flow/robustness-cmd` 的子集——这样异常覆盖由 Oracle B3 门裁决（非 0 / 通过数下降即打回），而非仅靠本节"我扫过了"的口述兜底。新增异常测试同样先红后绿（铁律 §1）。
- **模型独立性升级（异模型可用时）**：基线是 builder ≠ verifier（不同子代理）；当 `docs/flow/cross-verify` 声明了适配器，应进一步做到 **builder 模型 ≠ verifier 模型**——把 verifier 换成异模型/外部 agent（盲点不重叠，更难被同源 sycophancy 骗过）+ **多 change 间轮换谁建谁验** + 喂裁决前**抹除 implementer framing**（别带「这应该没问题」）。→ 用 Skill 工具加载 `cross-verify`（opt-in 多轮闭环；不可用则降级回本基线并显式告知）。

## 危险信号（出现即停 / 回退重来）

- 先写了实现才想起测试 / 测试一写就过、从没见它红过。
- 「这次特殊，因为……」「就破例这一次」「重要的是精神不是仪式」。
- 「旧代码留着当参考」「照着旧实现改改测试」。
- 把测试 `skip`/注释/弱化断言，或把期望值改成实际输出让它绿。
- verifier「没找到问题」就当通过；或 builder 顺手扮 verifier。
- 完成话术里出现 `should` / `seems` / `大概没问题` / 「同义改写了一遍当作覆盖」。

任意一条命中 → 删掉本轮代码，从 RED 重来；不是修补，是重来。

## 完成判定 = verify 通过 + 证据（不是你说了算）

builder 自测绿后，**加载 `verify` 技能**按档位跑项目真实 build/test 命令，并在同一轮贴出**新鲜输出**。完成由 verify + Stop hook 的 Oracle（`docs/flow/verify-cmd` 验证门 + `docs/flow/robustness-cmd` B3 健壮性门）裁决，不由叙述裁决。没有新鲜的通过输出 = 未完成（红线 §1）。

**封堵自报通过**：不信任子代理「我跑了，过了」的口头汇报——要它回传可核实的**命令 + 新鲜 stdout/退出码**。同义改写一句「测过了」不算证据。

**完成声明须可核实角色分离**：附独立 verifier 子代理的结论——**扫了什么**（边界/并发/错误路径/对照的反模式分区/做没做 mutation 抽查）+ **裁决**（发现的真 bug，或「无发现」）。只说「verifier 看过了」不算；hook 无法内省子代理是否真跑，靠这条可审计要求兜底（红线 §2/§3）。

## checklist（声明完成前逐条核）

- [ ] 每个新行为都有一条**先红后绿**的测试，且我亲眼见过它红。
- [ ] 没有后补测试、没有照旧实现改编的测试。
- [ ] builder 全程没改测试/断言/CI 去变绿。
- [ ] verify 跑了真实命令，新鲜输出已贴、退出码 0。
- [ ] verifier 是独立子代理，结论（扫了什么+裁决）已附。
- [ ] verifier 做过 mutation 抽查，确认测试真能失败。
- [ ] plan 的 MUST 异常场景已逐条落成测试并纳入 robustness-cmd 子集（受 Oracle B3 守护），非仅文本声明。
- [ ] 修 bug 时已查同文件/同模块的同类 bug。

## 红线（反合理化）

> 违反规则字面 = 违反规则精神。任何「我遵守了精神」的解释不成立。

| 合理化借口 | 实际规则 |
|---|---|
| 这改动很小不用写测试 | 改动大小不改变铁律：先写会失败的测试，再实现到通过。简单代码也出 bug，测试只要几十秒。 |
| 我先实现，之后补上测试 | 后补测试立即变绿、证明不了任何东西，且只测「我写了啥」而非「需求要啥」=作弊（铁律 §2）。 |
| 旧实现先留着当参考 | 你会照它改编，本质就是后补测试。删掉，从测试重新实现（铁律 §3）。 |
| 测试挂了先注释/skip 让进度往前 | 禁止注释/skip/弱化测试。挂了就是未通过，修实现而非改测试（红线 §1）。 |
| builder 顺手改下断言让它过 | builder 不得改测试/断言/CI；改断言属 verifier 域，违反权责分离（红线 §2）。 |
| 我把逻辑同义改写一遍，等于覆盖了 | 同义改写不是测试，不产生独立失败信号；要有能真正变红的断言。 |
| verifier 没找到问题就算通过 | 通过由真实 verify 命令的新鲜输出裁决，不由「没找到」推定；且 verifier 须做 mutation 抽查。 |
| 子代理回报说测过了，那就是过了 | 不信自报；要可核实的命令+新鲜 stdout+退出码，否则等于没验。 |
| 这个 bug 不在本次 task 范围内先放着 | 修一个 bug 时顺手查同文件/同模块同类 bug（红线 §4）。 |
| verifier 我 builder 顺便扮一下就行 | builder 与 verifier 必须是不同子代理；完成声明须附 verifier 独立结论，不可自演（红线 §2）。 |

完成后 → R1+ 加载 `document` 出交付物；撞上学习信号（连续失败/升维移交/返工/档位被推翻/verifier 抓到真 bug）则下一步用 Skill 工具加载 `harvest`。

遵循 `flow` 技能的质量红线。
