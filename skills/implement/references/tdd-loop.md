# TDD 循环纪律（RED-GREEN-REFACTOR）

> SKILL.md 铁律 §1–§3 的展开。本文件讲**每一步该亲眼看到什么**，以及为什么「补写测试」「拿旧实现当参考」都算作弊。implement 只管单个 change，按本节奏推进。

## 三步，顺序不可改

**RED — 写一个会失败的测试。**
- 先描述「需求要什么」，落成一条断言。
- 跑它，**亲眼看它失败**。看的不只是「红」，还要确认**失败原因正确**——是断言不满足而红，不是导入报错、拼写错、桩缺失而红。原因不对的红 = 假红，不算 RED。
- 还没失败过的测试，不是测试。

**GREEN — 写最少的代码让它通过。**
- 只写让当前这条测试变绿所需的代码，不顺手实现下一步。
- 跑全量测试，确认新绿、旧的没被你改红。

**REFACTOR — 测试保持绿的前提下清理。**
- 去重、改名、抽函数；每改一小步重跑测试。
- 一旦变红，回退到上一个绿点，别在红着的状态上继续重构。

一次只推进**一个最小可测行为**。贪多 = RED 里塞了多个失败原因，定位变慢，纪律垮掉。

## 为什么「后补测试」=作弊（三重理由）

1. **立即通过的测试什么也证明不了。** 你照着已写好的实现补测试，它一写就绿——这只证明「测试和实现长得一样」，没证明「实现满足需求」。
2. **它测的是实现，不是需求。** 后补测试受你实现的偏见污染：你测的是「我写了什么」，而 RED 优先逼你回答「应该做什么」。两者经常不是一回事。
3. **它发现不了你漏掉的边界。** 后补只会覆盖你**记得**的情况；先写测试才会逼你在实现前**发现**边界。

## 为什么「旧实现当参考」=作弊

重写、迁移、换框架时，最大的诱惑是「旧代码先留着对照」。

> 留作参考，你就会去改编它；那就是后补测试。删除就是删除——不保留、不在写测试时看它、不照着改。从测试出发，重新实现。句号。

正确做法：先把旧实现的**外部契约/行为**用测试钉下来（黑盒，不看内部），删掉旧实现，再从这些测试 GREEN 出新实现。

## 正例

> 要给 `parsePrice("$1,200.50")` 加千分位支持。
> 1. RED：写 `expect(parsePrice("$1,200.50")).toBe(1200.5)`，跑，红——当前返回 `NaN`，失败原因正确。
> 2. GREEN：在解析前 `replace(/,/g, "")`，跑，绿；旧用例仍绿。
> 3. REFACTOR：把清洗逻辑抽成 `stripFormatting`，重跑全绿。
> 4. 边界补红：`"$-1,000"`、`""`、`"1,2,3"` 各写一条，逐个红→绿。

## 反例（命中危险信号，删了重来）

> 「这个工具函数就三行，我先写完，等下统一补测试。」
> → 实现先落地，测试后补、一写就绿。它证明不了正确性，也漏掉了 `""` 和负数。这是铁律 §2 的典型违反：**删掉刚写的实现，从 RED 重来**，不是「补个测试缝合一下」。

> 「旧的 `LegacyParser` 我先注释留着，照它把新解析器写出来再说。」
> → 照旧实现写 = 后补测试的变体（铁律 §3）。删旧实现，先用黑盒测试钉契约，再重新实现。

## 自检：我真的在做 TDD 吗

- 我能指出每条测试**第一次变红**的那一刻吗？指不出 = 没做。
- 把实现随手改坏一处，对应测试会变红吗？不会 = 假测试（verifier 的 mutation 抽查就查这个）。
- 我有没有在「就这一次」「情况不同」的话术里给自己开后门？有 = 危险信号，删码重来。
