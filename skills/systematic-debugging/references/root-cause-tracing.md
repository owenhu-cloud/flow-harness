# 反向追数据流定位根因

调试的阶段二核心动作：从**报错点**出发，沿数据/控制流**反向**走到污染的源头，而不是在报错点就地打补丁。报错点几乎总是症状暴露的地方，不是 bug 产生的地方。

## 方法

1. **从异常值反推。** 报错那行拿到的是什么坏值（null / 越界 / 类型错 / 脏状态）？记下「期望 vs 实际」。
2. **问：这个坏值从哪来？** 上溯一层调用/赋值，定位它进入当前作用域的入口。
3. **逐层重复**，直到找到第一处「值在这里变坏 / 假设在这里被打破」的点 —— 那才是根因层。
4. **二分定位（多组件系统）。** 在每个组件边界打诊断日志，看坏值是从哪个边界开始出现，把搜索空间对半砍。
5. **坐实因果。** 在根因层证明「改这里 → 症状变 / 消失」，写出「X 导致 Y 因为 Z」。改不动症状 = 还没到根因，继续上溯。

## 铁律呼应

- 一次只动一处（SKILL 铁律 §2）：追因时也是，加一处日志/断点看一处，别同时翻三层。
- 找到根因前不写修复（SKILL 铁律 §1）：追到源头之前的任何「先挡一下」都是症状补丁。

## 危险信号

- 在报错行加 `if (x == null) return` 而不问 x 为何为 null —— 典型症状补丁，根因往上游漂。
- 「日志太多看不过来」就跳过边界追踪，凭直觉猜哪段坏 —— 回到二分，让证据说话。
- 上溯两层就「差不多就是这了」停手 —— 没坐实「改这里症状就变」之前不算定位。

## 正例 / 反例

- 反例：接口返回 500，直接在 controller 兜 try/catch 返默认值 → 上游脏数据继续扩散，下次别处再炸。
- 正例：500 的栈指向反序列化越界 → 上溯发现某上游写入跳过了 schema 校验 → 根因是写入层缺校验，修在写入层（必要时配合 defense-in-depth 两层都补）。
