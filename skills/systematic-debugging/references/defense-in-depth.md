# 纵深防御（别只在一层打补丁）

阶段四防回归时的判断：一个坏值/坏输入能造成 bug，往往是因为它穿过了**多层都没人拦**。只在报错那一层加个判断，治得了这一次，治不了下一个从别的入口进来的同类输入。纵深防御 = 在数据流的**多个合适层**各自校验，任一层失守仍有兜底。

## 原则

- **修在根因层 + 在边界层加守卫。** 根因层（如写入处缺校验）必须修；同时在信任边界（入口/反序列化/持久化前）补一道校验，挡住未来同类脏数据。
- **不是到处加 if。** 层要选在**有意义的信任边界**上（外部输入入口、跨模块契约、写库前、消费前），不是在调用链每一格都塞判断——那是噪声不是防御。
- **每层失败都要显式、可观测。** 拦下来就明确报错/记日志，不静默吞掉变成新的隐形 bug。

## 与「最小修复」的边界（重要）

SKILL 铁律 §4 要最小修复，本文要多层防御 —— 不冲突，靠根因区分：

- 若坏值只可能从**一个**源头产生 → 修那一处即可，别为「以防万一」全链加判断（那叫过度防御/噪声）。
- 若同类坏值可从**多个入口**进入、或失守代价高（数据损坏/安全/资金）→ 才在多层设防。
- 拿不准时回阶段二的数据流图：数出有几个真实入口，按入口数决定设几层，不拍脑袋。

## 危险信号

- 在最末端 catch 一把兜底默认值，上游脏数据照样扩散 —— 这是症状补丁不是防御。
- 「在这层挡一下就行」却没问同类输入还能从哪进来 —— 漏防。
- 反向：每个函数开头都复制粘贴同一串校验 —— 过度，抽成边界处一道即可。

## 正例 / 反例

- 反例：反序列化越界，只在崩溃的消费点加 `if (idx < len)`。下个消费点换人写又炸。
- 正例：写入层补 schema 校验（根因层）+ 消费前对不变量断言（边界层），两层都显式失败并记日志；新增测试覆盖两层各自被坏输入触发。

回到 `SKILL.md` 阶段四。先定根因层必修，再按入口数决定要不要加边界层，别一把 if 糊全链。
