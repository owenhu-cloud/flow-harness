# systematic-debugging — 四阶段根因调试

**适用档位：R1–R3**（R0 单行一键回滚的小错可直接改；只要「改了又坏 / 不知道为什么 / 涉及多组件」就升进本技能）。在 Flow 流程 `brainstorm → plan → implement → verify → document` 中，本技能是 **implement 的前置根因环节**：只负责**阶段一/二——稳定复现 + 定位根因 + 写出会失败的测试**；**阶段三（实现修复）交回 `implement`** 走 builder/verifier 对抗，完成仍由 `verify` + Stop hook Oracle 裁决。本技能不另立完成判定，只保证「带着真根因和失败测试」再去修。

动手前先读 `docs/flow/project.md`（项目画像）：用其中的真实复现/构建/测试命令与项目特异反模式，别套通用规范。无画像 → 先用 `profile` 固化。

## 铁律（iron-laws，不可违反）

1. **无根因不提修复。** 没完成阶段一（稳定复现 + 定位到「为什么会发生」）之前，不许写任何修复代码、不许提修复方案。说得出「X 导致 Y，因为 Z」并能指到具体那行/那个状态，才算有根因。
2. **一次只改一个变量。** 每次验证只动一处，看清因果再动下一处。「多处一起改看哪个生效」= 自毁因果链，禁止。
3. **修复前先有失败测试。** 先写一条能稳定复现该 bug 的失败测试（自动化或脚本），亲眼见它红，再修到绿。没有这条测试 = 修复不会留存、回归无人看守（与 `implement` 铁律一致）。
4. **只做命中根因的最小修复。** 不顺手重构、不「连这个一起改好」。一次修复对应一个根因。
5. **症状消失 ≠ 修好。** 完成由 `verify` 的新鲜输出 + Stop hook Oracle 裁决，不由「现在不报错了」裁决（见「完成判定」）。

## 四阶段（顺序不可跳）

1. **复现** —— 完整读错误（栈/行号/错误码，别只看最后一行）；做出**稳定可复现**的步骤并记录；查最近改动（`git diff`、依赖、配置）。多组件系统：在每个组件边界加诊断日志，先定位是哪一段出错，再谈修复。
2. **定位根因** —— 从报错点**反向追数据流**到源头（详见 `references/root-cause-tracing.md`）；在同代码库找一个**能工作**的同类例子，逐行对照列出全部差异（依赖、配置、隐含假设）。
3. **最小修复** —— 明确写出**单一假设**：「我认为 X 导致 Y，因为 Z」；先写失败测试（铁律 §3），再做命中根因的最小改动（铁律 §4）；一次一变量验证（铁律 §2）。
4. **防回归** —— 失败测试转绿且不破坏其他测试；查同文件/同模块的**同类 bug**一起补（红线 §4）；必要时按 `references/defense-in-depth.md` 评估是否该补在多层而非单点。

## 卡住升维（与 `flow` 升维表呼应，不另立机制）

- **2 次修复失败 → 重收证据**：停止改代码，回阶段一。已有的复现/日志多半不够，补诊断、重读完整报错，别在同一假设上调参。
- **3 次修复失败 → 升维质疑架构**：这通常不是局部 bug，而是设计/契约问题。按 `flow` 升维表列 **3 个根本不同的假设**逐一证伪；仍不收敛则质疑「这个模块/契约本身是否成立」，结构化移交。
- 同一处反复修却不收敛 → 强制提一个 180° 反向假设（「也许根因根本不在我一直盯的这层」）。

## 危险信号（出现即停 / 回退到阶段一）

- 「先临时修一下，回头再查」「改个 X 看看灵不灵」「这几处一起改」。
- 「我不完全懂为什么，但这样应该能好。」
- 已经试了 ≥2 次修复都没成 —— 别试第 3 个补丁，回去重收证据。
- 每修一处就冒出新问题别处 —— 是在追症状不是追根因。
- 完成话术出现 `should` / `seems` / `大概 / 应该没问题` / 「现在跑一下不报错了」当通过。
- 用 `sleep N` 硬等来「让它过」—— 见 `references/condition-based-waiting.md`，禁止。

## 完成判定 = verify 通过 + 证据（不是「不报错了」）

写完修复，**加载 `verify` 技能**按档位跑项目真实 build/test 命令，同一轮贴**新鲜输出**。完成由 `verify` + Stop hook Oracle（`docs/flow/verify-cmd`）裁决。

- **封堵自报通过**：不信任子代理「我跑了，过了」的口头汇报，要可核实的**命令 + 新鲜 stdout + 退出码**。
- **封堵同义改写**：把日志措辞改一改、把断言改成实际输出让它绿 = 作弊，等同删测试变绿。
- **复现测试必须真红过**：那条防回归测试要在修复前亲眼见它失败——没红过的测试证明不了 bug 被修好（mutation 心智）。

## checklist（声明修好前逐条核）

- [ ] 有稳定复现步骤，且完整读过报错（非只看最后一行）。
- [ ] 写得出「X 导致 Y 因为 Z」并能指到具体代码/状态（阶段一过关）。
- [ ] 有一条**先红后绿**的防回归测试，且亲眼见它红过。
- [ ] 本次只做了命中根因的最小修复，没顺手重构。
- [ ] 已查同文件/同模块的同类 bug 一并处理。
- [ ] `verify` 跑了真实命令，新鲜输出已贴、退出码 0。
- [ ] 没用 `sleep N` 硬等；等待都改成轮询条件（condition-based-waiting）。

## 红线（反合理化）

> 违反规则字面 = 违反规则精神。任何「我遵守了精神」的解释不成立。

| 合理化借口 | 实际规则 |
|---|---|
| 这是个简单 bug，跳过流程直接修 | 简单 bug 也要有根因，四阶段走完只要几分钟；省下的是反复返工（铁律 §1）。 |
| 紧急，没时间做系统调查 | 系统调试比反复瞎猜更快收敛。越急越要一次修对，第一个补丁会定下后续轨迹。 |
| 先改这个看看灵不灵，灵了再说 | 「试一下」就是无根因猜修；第一个修复定轨迹，先把根因坐实（铁律 §1）。 |
| 我看到问题在哪了，直接改 | 你看到的是症状，症状 ≠ 根因；要追到数据流源头才算定位。 |
| 几个地方一起改，省得来回跑 | 一次一变量；多处齐改无法隔离哪个生效，还会引入新 bug（铁律 §2）。 |
| 修复确认好了之后再补测试 | 没有先红的测试，修复不留存、回归无人看守；测试必须先于修复（铁律 §3）。 |
| 那个能工作的例子我大概扫一眼就行 | 部分理解正是 bug 来源；逐行对照列全差异，别跳读。 |
| 已经试了两三次了，再换个补丁试试 | ≥2 次失败=证据不够，回阶段一重收；3 次=质疑架构，不是第 4 个补丁。 |
| 现在跑一下不报错了，就是修好了 | 症状消失不等于修好；完成由 verify 新鲜输出 + Oracle 裁决（铁律 §5）。 |
| 子代理回报说修好测过了，那就过了 | 不信自报；要可核实的命令+新鲜 stdout+退出码，否则等于没验。 |
| 加个 sleep 等它准备好就稳定了 | 硬等是在掩盖竞态根因；改成轮询条件（见 condition-based-waiting）。 |
| 在出错这层加个判断挡一下就行 | 单点补丁常漏同类输入；按根因评估是否该多层防御（defense-in-depth）。 |
