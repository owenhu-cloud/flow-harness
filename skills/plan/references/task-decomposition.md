# 任务拆解纪律（plan 的核心手艺）

写 `tasks.md` 前先读这份。目标：让每条任务**小、有序、可独立验证、路径明确**，使 implement 能逐条 builder/verifier 推进，verify 能逐条裁决。

## 四条判据（每条 task 都要满足）

1. **小**：一条任务对应一个逻辑单元、一次可审查的改动。大到「写不出单一验证方式」就拆。
2. **有序**：能线性推进；后一条不依赖前一条的**未落地假设**（接口、类型、数据形状必须在前置任务里已定义，不能"实现时再说"）。
3. **可独立验证**：「验证方式」是可运行命令或可观测断言，单独跑这条就能判真假。
4. **路径明确**：写出具体文件 / 函数 / 接口名；引用到的类型、方法必须在某条任务里被定义过——孤儿引用 = 缺口。

## 切口怎么选：按复杂度 + SOLID（不是凭感觉切）

上面四条判据说的是"切完要满足什么"，这节说"**从哪切**"——拆错缝会让任务间隐藏耦合、无法独立验证、无法分派独立 agent。

### 第一步：按复杂度定"拆不拆"（反过度设计，和判据同等重要）

- **R0 / R1 → 直接实现，不拆、不套设计模式。** 允许"直接做"的硬条件（**全部满足**才算）：单文件/单模块 · 无新增公共契约/接口 · 不触 schema/迁移/认证/支付/生产数据（即不命中 `flow` 档位地板）· 一条验证命令即可裁决。给几十行函数套 SOLID/工厂/策略/分层是**过度设计**，本身是反模式（见 `implement/references/antipatterns.md`）。「简单项目直接实现」是纪律，不是偷懒。
- **R2 / R3（多模块/跨系统/需设计）→ 沿 SOLID 缝拆。** 复杂度高才值得拆；拆的目的是让每块**单一职责、靠接口解耦、可独立验证、可分派独立 agent**。

判档复用 `flow` 技能的四维 rubric，不另起一套。注意方向：**"拿不准/未知度高"在 flow 是升档信号（不是压低借口）**——该先 `research` 或升档，不是借"拿不准"留在 R1 跳过拆解。真正该防的是**过度拆分**：把本来内聚、只能靠共享内部细节协作的逻辑切碎成一堆互相引用的小任务，和拆不够一样有害（见第五问"自检合并"）。

### 第二步：复杂任务用 SOLID 找缝（每条原则指一种切口）

| 原则 | 它指出的切口 | 落到 task / agent |
|---|---|---|
| **SRP** 单一职责 | 一条任务只为**一个变更理由**负责 | 一条 task = 一个职责；掺了两个理由 → 拆两条。决定"切多大" |
| **DIP** 依赖倒置 | 任务间靠**接口/契约**连，不靠具体实现 | 接口在前置任务**先定义**（正是上面判据2），下游依赖接口而非上游实现 → 两条可独立 agent 落地、独立验证。决定"缝在哪" |
| **ISP** 接口隔离 | 别让一条任务被迫依赖它用不到的大接口 | 切小接口，下游只依赖自己要的那部分 |
| **OCP** 开闭 | 预期会扩展的点，切成"加新文件"而非"改老文件" | 扩展型任务=新增实现+注册，避免多 agent 改同一文件冲突 |
| **LSP** 里氏替换 | 同族实现走统一契约、可互换 | 多实现任务共享一份**契约测试** |

最吃劲的是 **SRP（切多大）+ DIP（缝在哪）**：SRP 定一条任务的边界，DIP 定两条任务之间用什么连——接口先定义、下游依赖接口，于是两条能独立落地且独立验证。这直接喂 `subagent-driven-development` 的串行派发。

### 切口提取流程（从需求 prose 走到 tasks，照走、不靠悟）

原则表是"映射"，这五问是"怎么从一段需求机械走到切口"：

1. **列变更理由**：这个 change 未来会因为哪些**相互独立的业务/技术理由**被改动？每个独立理由 = 一条候选任务的边界（SRP）。**列不出 ≥2 个独立理由 → 它就是一条任务，别拆。**
2. **找跨候选依赖**：候选任务之间互相用到的**数据形状 / 调用**是哪些？把它们提成**显式契约**（接口/类型/数据格式），放进一条"先定义契约"的前置任务（DIP）。下游只依赖契约、不依赖上游实现。
3. **判契约该不该抽**：一个接口只有满足"**≥2 实现或消费者 / 跨外部边界（网络·进程·DB）/ 需要测试缝**"之一才抽；否则直接用具体实现——**为单一实现造接口是过度设计**（OCP 戏剧化），别为抽象而抽象。
4. **定顺序与派发**：按形状（分层/管道/独立特性）排序；契约任务永远排在依赖它的任务**之前**；全程**串行**派（subagent-dev 铁律1，防写冲突）。
5. **自检合并**：若两条切出的任务仍需**共享大量内部细节**才能各自完成（而非仅通过契约通信）→ 切错缝了，**合并回一条**。过度拆分和拆不够一样有害。

R2/R3 把这趟的结论（为什么拆这几条、契约是什么）落一句进 `design.md` 的 Decisions，使切口**可审**——不是让 agent 自己心里有数就算。

### 第三步：任务形状 → 拆分 / 派发策略

| 形状 | 怎么切 | 派发（subagent-driven-development） |
|---|---|---|
| **分层/依赖链**（A 定义 → B 使用） | 按层切，层间接口在上游任务**先落盘**（DIP） | 串行：上游接口落定后才派下游 |
| **独立特性/限界上下文** | 按特性切，缝落在 bounded-context 边界、互不碰同一文件 | 仍串行派（铁律1 防写冲突），但缝独立 → 评审简单、返工不扩散 |
| **管道/变换**（stage1 → stage2） | 按 stage 切，stage 间**数据契约**先定义 | 串行，按 stage 顺序 |
| **横切关注点**（日志/鉴权/校验/错误处理） | **不单独切成任务**——作为每条任务内部的约束（`project.md` 反模式 + design 的 Robustness-Cases），否则到处改、多 agent 冲突 | 不派独立 agent |

### 一个对比例（同类需求，简单 vs 复杂走不同路）

- **简单（R1）**：「给 config 加一个 `timeout` 字段」→ 一条任务直接改，**别拆、别套模式**。套个策略工厂就是空架子。
- **复杂（R3）**：「支付接入三家网关」→ 走五问：变更理由有"网关协议各异(×3)+对账编排"四类（SRP）；跨依赖是各家共用的 `Gateway` 抽象（DIP，满足"≥2 实现"故值得抽）。切成**五条**：①定义 `Gateway` 接口 + 契约测试（前置）→ ②③④ 三家各自实现依赖①（OCP：加第四家=加新文件不改老的）→ ⑤编排/对账依赖接口、不依赖具体家。五条各自可独立验证；**串行派发**（铁律1）但可分派**不同模型**——①接口设计派最强档、②③④机械实现派预算档（见 subagent-driven-development 模型分级）。

## 验证方式怎么写（反 reward-hacking）

「验证方式」是 implement→verify 的契约，不是装饰。

- **可执行**：`go test ./auth -run TestX`、`curl … 返回 200 且 body.field==…`、`二分日志出现 "migrated N rows"`。
- **禁措辞**：「手测一下」「看下效果」「应该没问题」「seems/should 大概对」——这些不可裁决，等于没写。
- **禁同义改写充数**：验证方式不能只是把任务标题换个说法重述一遍（「实现 X」配「验证 X 已实现」）。必须指向**独立于实现**的外部信号。
- **映射到测试或可观测结果**：每条任务要么能写一个测试覆盖它，要么产出一个可观测变化（接口响应、日志、数据状态）。两者都没有 → 任务定义不完整。

## 占位语黑名单（出现即缺陷）

`TBD` · `TODO` · `待定` · `加适当的错误处理` · `处理好边界情况` · `补充测试`（不说补什么）· `优化性能`（不给指标）。

这些不是任务，是把思考推迟到 implement。要么现在写出具体行为，要么拆成一条有明确验证方式的独立任务。

## 更多正反例

### 拆得太粗
```
坏:  - [ ] 1 做好支付模块 — 验证方式：能付款
```
问题：影响面巨大、无路径、验证不可裁决。

### 拆得合适
```
好:  - [ ] 1.1 在 pay/order.go 定义 Order{ID,Amount,Status} 与状态机 pending→paid→refunded
        — 验证方式：go test ./pay -run TestOrderTransition，非法跃迁返回 err 全绿
     - [ ] 1.2 在 pay/gateway.go 加 Charge(orderID) 调用 stub 网关；失败置 Order.Status 不变
        — 验证方式：go test ./pay -run TestChargeIdempotent，重复 Charge 不重复扣款
     - [ ] 1.3 在 api/pay_handler.go 暴露 POST /pay；校验金额>0
        — 验证方式：curl -d amount=0 返回 400；amount=100 返回 200 且 status=paid
```
注意：1.2 用到的 `Order.Status` 在 1.1 已定义（无孤儿引用）；三条各自可单测；命名一致。

### 隐藏依赖（要消除）
```
坏:  - [ ] 2.1 加缓存层
     - [ ] 2.2 用 2.1 的缓存键格式做失效 — 但 2.1 没定义键格式
```
修法：把「缓存键格式」作为 2.1 的显式产出写进任务描述，2.2 才有可依赖的落地物。

## 敏感面提醒

若任务触及 DB 迁移 / 鉴权 / 发布 / 计费（见 flow 技能档位地板），design.md 的 Risks-Tradeoffs 必须写明**回滚策略**，对应任务的验证方式要包含可回滚证明（如「迁移可 down 且数据无损」），不能只验「正向跑通」。
