# 任务拆解纪律（plan 的核心手艺）

写 `tasks.md` 前先读这份。目标：让每条任务**小、有序、可独立验证、路径明确**，使 implement 能逐条 builder/verifier 推进，verify 能逐条裁决。

## 四条判据（每条 task 都要满足）

1. **小**：一条任务对应一个逻辑单元、一次可审查的改动。大到「写不出单一验证方式」就拆。
2. **有序**：能线性推进；后一条不依赖前一条的**未落地假设**（接口、类型、数据形状必须在前置任务里已定义，不能"实现时再说"）。
3. **可独立验证**：「验证方式」是可运行命令或可观测断言，单独跑这条就能判真假。
4. **路径明确**：写出具体文件 / 函数 / 接口名；引用到的类型、方法必须在某条任务里被定义过——孤儿引用 = 缺口。

## 验证方式怎么写（反 reward-hacking）

「验证方式」是 implement→verify 的契约，不是装饰。

- **可执行**：`go test ./auth -run TestX`、`curl … 返回 200 且 body.field==…`、`二分日志出现 "migrated N rows"`。
- **禁措辞**：「手测一下」「看下效果」「应该没问题」「seems/should 大概对」——这些不可裁决，等于没写。
- **禁同义改写充数**：验证方式不能只是把任务标题换个说法重述一遍（「实现 X」配「验证 X 已实现」）。必须指向**独立于实现**的外部信号。
- **映射到测试或可观测结果**：每条任务要么能写一个测试覆盖它，要么产出一个可观测变化（接口响应、日志、数据状态）。两者都没有 → 任务定义不完整。

## 占位语黑名单（出现即缺陷）

`TBD` · `TODO` · `待定` · `加适当的错误处理` · `处理好边界情况` · `补充测试`（不说补什么）· `优化性能`（不给指标）。

这些不是任务，是把思考推迟到 implement。要么现在写出具体行为，要么拆成一条有明确验证方式的独立任务。

## 更多正反例

### 拆得太粗
```
坏:  - [ ] 1 做好支付模块 — 验证方式：能付款
```
问题：影响面巨大、无路径、验证不可裁决。

### 拆得合适
```
好:  - [ ] 1.1 在 pay/order.go 定义 Order{ID,Amount,Status} 与状态机 pending→paid→refunded
        — 验证方式：go test ./pay -run TestOrderTransition，非法跃迁返回 err 全绿
     - [ ] 1.2 在 pay/gateway.go 加 Charge(orderID) 调用 stub 网关；失败置 Order.Status 不变
        — 验证方式：go test ./pay -run TestChargeIdempotent，重复 Charge 不重复扣款
     - [ ] 1.3 在 api/pay_handler.go 暴露 POST /pay；校验金额>0
        — 验证方式：curl -d amount=0 返回 400；amount=100 返回 200 且 status=paid
```
注意：1.2 用到的 `Order.Status` 在 1.1 已定义（无孤儿引用）；三条各自可单测；命名一致。

### 隐藏依赖（要消除）
```
坏:  - [ ] 2.1 加缓存层
     - [ ] 2.2 用 2.1 的缓存键格式做失效 — 但 2.1 没定义键格式
```
修法：把「缓存键格式」作为 2.1 的显式产出写进任务描述，2.2 才有可依赖的落地物。

## 敏感面提醒

若任务触及 DB 迁移 / 鉴权 / 发布 / 计费（见 flow 技能档位地板），design.md 的 Risks-Tradeoffs 必须写明**回滚策略**，对应任务的验证方式要包含可回滚证明（如「迁移可 down 且数据无损」），不能只验「正向跑通」。
