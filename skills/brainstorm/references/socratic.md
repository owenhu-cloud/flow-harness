# 提问与候选方案打磨手册

主文件 `SKILL.md` 的渐进披露：当意图不清或候选方案趋同时，按此操作。

## 苏格拉底式追问

铁则：**一次只问一个问题**；**多选题优于开放题**（降低用户负担、逼出隐含取舍）。

聚焦三类，按缺口优先：
1. **目的**：要解决的真痛点是什么？这是真问题还是 XY 问题（用户在问解法，但真痛点在上游）？
2. **约束**：性能 / 兼容 / 期限 / 不可改的接口 / 数据规模？
3. **成功标准**：怎样算成功，且**可观测**？「更好/更快」不算，要「P99 < X」「不再需要手动 Y」。

何时停止追问：目的、约束、成功标准三项都清楚后立即停，转去列候选。别为完美信息无限追问——拿到能区分方案的信息就够。

XY 问题嗅探：用户直接要某个具体技术解法时，先反问「你想用它达成什么」。若上游有更简单的路，方向就变了。

## 多系统先拆分

若需求横跨多个相互独立的子系统：**立即指出，先拆**。不要花时间用问题去细化一个本该先拆分的项目——那是在错误的粒度上对齐。拆完后对每个子项目分别判断档位。

## 候选方案要「本质不同」

不同 = 落在不同的**取舍轴**上，而非同一思路调参数。常见取舍轴：

- 简单 vs 可扩展（先扛住 vs 扛长远）
- 读优化 vs 写优化
- 自建 vs 复用现成（控制力 vs 上手速度）
- 同步 vs 异步（一致性 vs 吞吐）
- 通用抽象 vs 针对性硬编（灵活 vs 当下最省）
- 集中 vs 分布（简单 vs 隔离/容错）

每个候选必须标三件事：
- **核心权衡**：它用什么换什么。
- **失败模式**：在什么条件下它先崩。
- **适用边界**：什么规模/场景之外不该选它。

自检：若两个候选的「核心权衡」写出来几乎一样 → 它们不是两个方案，合并，再去另一条轴上找第二个。

## 收敛

收敛 = 选 1 个方向 + 明确写出**为什么不选其它**（放弃理由本身就是给 plan 和未来的你留的决策记录）。
收敛由用户确认，不由你宣布。复述格式：「选定 X，因为 …；放弃 Y（因 …）、Z（因 …）。这个方向和取舍，对吗？」
