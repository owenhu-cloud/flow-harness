# brainstorm — 对齐方向（R2/R3）

**位置**：`brainstorm → plan → implement → verify → document` 的第 1 站。
**档位**：仅 R2/R3。R0/R1 方向自明，跳过本技能，直接按 `flow` 给的轻流程做。
**唯一产出**：被用户明确点头的「方向共识」——选定方向 + 关键权衡 + 放弃的备选及原因。不写 tasks、不定文件路径、不碰代码。

## 铁律（IRON LAWS，违反即作废）

1. **方向未经用户确认前，零实现动作**：不加载 `plan`/`implement`，不建目录、不写脚手架、不改任何文件。
2. **至少 2 个本质不同的候选**：必须是不同思路（不同架构/不同取舍轴），不是同一方案的参数变体。只想出 1 个 = 还没 brainstorm。
3. **苏格拉底优先于方案**：意图不清时先追问澄清，绝不拿假设直接灌方案。
4. **发散调研外包**：检索/调研派 `research` 子代理（`Task` 工具），主线程只留人机对话与收敛，守护主上下文。
5. **收敛靠人，不靠自我宣布**：方向是否定了，由用户的明确确认裁决，不是你说「方向应该挺清楚了」。

## 硬门（HARD GATE）

> **在「方向共识 + 关键权衡」被用户明确确认之前，禁止进入 plan、禁止改文件。**
> 自检一句：「方向定了吗？用户点头了吗？」任一为否 → 留在本阶段。

## 合理化借口 → 实际规则（红线表）

| 你在压力下会对自己说的原话 | 实际规则 |
|---|---|
| 「需求挺简单的，跳过对齐直接干」 | 越简单越是未检验假设最多的地方；R2/R3 不跳。简单到不需对齐 = 它是 R0/R1，该走 `flow` 重判档位而非省略。 |
| 「我先搭个架子/建个目录，边写边讨论」 | 用执行逃避对齐。零文件改动，先把方向说清。 |
| 「我想到一个好方案，够了」 | 单方案不构成权衡。强制再产 ≥1 个本质不同候选，否则你在替用户做未公开的取舍。 |
| 「用户没明说，我按最可能的理解推进」 | 假设即风险。把假设作为问题抛回，让用户确认，而不是默默替他选。 |
| 「方向应该 / 大概 / 看起来定了，进 plan 吧」 | should/seems/大概=未确认。必须有用户对「这个方向 + 这些权衡」的显式肯定。 |
| 「这几个候选其实差不多，随便选一个」 | 差不多 = 你没找到真正的取舍轴。要么挖出区别，要么合并后再找第二条不同思路。 |
| 「需求里有好几个独立系统，我一起 brainstorm」 | 立即停。先拆成独立子项目，别用问题去细化一个本该先拆分的项目。 |

## 危险信号（出现即停 / 回退）

- 手已经伸向 `Write`/`Edit`/建目录 → **停**，回到对话。
- 候选列表里全是同一思路的变体 → **回退**，换取舍轴重列。
- 用户给的需求横跨多个独立子系统 → **停**，先拆分再谈。
- 你开始写文件路径、函数名、tasks → 越界进 plan，**收回**。
- 连追问 2 轮意图仍不收敛、或新颖度高心里没底 → 先派 `research`，别硬编方案。

## 决策流程

```
理解意图 → 真问题还是 XY 问题？
  ├─ 不清楚 → 苏格拉底式追问（一次一问，多选优于开放），先澄清
  └─ 清楚 → 列 2–4 个本质不同候选（不同取舍轴，非参数变体）
        → 每个标：核心权衡 / 失败模式 / 适用边界
        → 收敛到 1 个选定方向 + 写明为何不选其它
        → 向用户复述「方向 + 关键权衡 + 放弃项」，取确认 ──HARD GATE──▶ plan
```

提问技巧、候选方案打磨、取舍轴清单 → 见 `references/socratic.md`。

## 澄清清单（逐条自查，缺口即追问）

> 这就是 Flow 的「规格澄清 / spec-clarify」——不另设技能：模糊需求在此用苏格拉底追问逼成可对齐的方向，歧义不清零不进 plan。

- 真正要解决的痛点是什么？成功长什么样（**可观测**，非「更好」）？
- 硬约束：性能 / 兼容 / 期限 / 不可改的接口？
- 显式**非目标**（这次不做什么）？
- 谁是用户、谁来维护、影响哪些上下游？

## 正反例

- ✅ 「方案 A 重读快写慢（列存），B 读写均衡但占空间（双写索引），C 简单但不扛量（单表）。你的查询是分析型还是事务型？据此我倾向 A，因为非目标里说了写入低频——这样选对吗？」→ 不同轴、点出非目标、抛回确认。
- ❌ 「我觉得用方案 A 挺好的，我先把表结构和目录建起来，你看看。」→ 单方案 + 越界动手，双重违反铁律 1、2。

## checklist（进 plan 前全勾）

- [ ] 已确认是 R2/R3（否则不该在这）。
- [ ] 列了 ≥2 个本质不同候选，各带权衡/失败模式/边界。
- [ ] 意图无残留歧义；关键假设已抛回用户确认。
- [ ] 收敛出 1 个方向，并写明放弃其它的理由。
- [ ] 用户对「方向 + 权衡」给了**显式**肯定（非你的揣测）。
- [ ] 全程零文件改动、零代码。
