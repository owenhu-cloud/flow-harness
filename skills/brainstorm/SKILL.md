# brainstorm — 对齐方向（R2/R3）

只产**共识与方向**，不写最终计划、不定文件路径、不碰代码。

## 硬门（HARD GATE，不可合理化）

> **在方向共识 + 关键权衡被用户明确确认之前，禁止加载 plan / implement 技能，禁止改任何文件。**
> 「我先搭个架子再讨论」「顺手把目录建了」都属违反——那是用执行逃避对齐。

判断是否还在 brainstorm：问自己「方向定了吗？用户点头了吗？」否 → 留在本阶段。

## 决策流程

```
理解意图 → 这是真问题还是 XY 问题？
  ├─ 不清楚 → 苏格拉底式追问，先澄清，别急着给方案
  └─ 清楚 → 列 2–4 个本质不同的候选方案（不是同一思路的参数变体）
        → 每个方案标：核心权衡 / 失败模式 / 适用边界
        → 收敛到 1 个选定方向 + 写明为什么不选其它
        → 向用户复述方向与权衡，取得确认 ──HARD GATE──▶ 进 plan
```

## 边界（与 plan 刻意分开）

- **做**：探问题本质、列候选方案、辨权衡、苏格拉底式追问澄清意图。
- **不做**：写 tasks、定路径、落设计——那是 plan 的事。

## 澄清清单（逐条自查，缺口即追问）

- 真正要解决的问题 / 痛点是什么？成功长什么样（可观测）？
- 硬约束：性能 / 兼容 / 期限 / 不可改的接口？
- 显式**非目标**（这次不做什么）？
- 谁是用户、谁来维护？影响哪些上下游？

## 守护主上下文

- 发散性检索/调研**派 `research` 子代理**（用 `Task` 工具），主线程只留人机对话与收敛。
- 不确定且新颖 → 先 research 再继续。
- 候选方案的细节论证让子代理回传摘要，主线程只审结论。

## 产出 → 交接

对话中达成的方向共识（含「选定方向 + 关键权衡 + 放弃的备选及原因」），必要时记入 `docs/flow/<change>/context.md`。
方向确认后 → 用 `Skill` 工具加载 `plan`。

遵循 `flow` 技能的质量红线。
