# subagent-driven-development — 串行单任务派发 + 双评审 + 模型分级

**适用档位：R2 / R3。** 在 Flow 流程 `brainstorm → plan → implement → verify → document` 中，本技能是 **implement 的编排外壳**：`plan` 产出的 `tasks.md` 有**一串**任务时，由本技能把它们**逐条**派给**独立子代理**落地，每条任务内部仍走 `implement` 的 TDD + 对抗证伪。

**与 implement 的分工**（别混）：`implement` 钉**单个 change** 的 builder/verifier 节奏；本技能钉**多个任务的编排**——谁来做、用什么模型、做完谁评审、不过怎么回退。一句话：implement 是「一条任务怎么做对」，本技能是「一串任务怎么逐条派出去不串味」。

开工前读 `docs/flow/project.md`（项目画像）：派发给子代理的 brief 必须把画像里的命令、风格约定、项目特异反模式**原文带进去**——子代理是 fresh 上下文，你不带它就不知道。无画像 → 先用 `profile` 固化。

**任务怎么切决定怎么派**：本技能消费 `plan` 按复杂度 + SOLID 切好的 tasks（见 `plan/references/task-decomposition.md` 的「切口怎么选」+「任务形状→派发策略」）。简单（R0/R1）本就一条任务、不进本技能直接 `implement`；复杂任务按形状派——分层/管道按依赖串行（上游接口先落再派下游）、独立特性缝独立但仍串行防写冲突、横切关注点不单独派。**别拿到一坨没沿 SOLID 切的任务硬派**——缝没切对，独立 agent 落地必然互相踩、评审无从独立裁决，先退回 `plan` 重切。

## 铁律（iron-laws，违反即作废）

1. **串行，一次一个任务。** 绝不同时派多个实现子代理跑不同任务——并发写同一代码库必然冲突、互相覆盖。前一个任务双评审全过、落盘，才派下一个。
2. **每个任务派 fresh 子代理。** 不复用上一个任务的子代理上下文——上文的细节会串进来污染本任务（reward-hacking 的温床：它会拿上个任务的"差不多"来糊本任务）。隔离是本技能的核心价值。
3. **子代理不读 plan 文件，你把任务全文喂给它。** 派发时把该任务的**完整规格文本 + 必要上下文**写进 brief，而不是丢一句"去看 tasks.md 第 3 条"。让它读文件会读串、读漏、自行扩范围。
4. **双评审顺序固定：spec 合规 → 代码质量。** spec 合规没过，绝不开始代码质量评审（spec 不对，质量再高也是错东西）。两道门都过，任务才算完。
5. **评审与实现角色分离，且评审不可自演。** 实现子代理不得评审自己的产出；spec 评审、质量评审各由独立子代理做。完成判定见下，不由实现者自报。
6. **BLOCKED/上报绝不"原样重试"。** 子代理卡住或回 BLOCKED，必须先定位根因再动作（见状态机），绝不在不做任何改变的情况下让同一模型重跑一遍碰运气。

派发 brief 模板、模型分级信号、实现者状态机（DONE/DONE_WITH_CONCERNS/NEEDS_CONTEXT/BLOCKED）的完整处置见 `references/dispatch-protocol.md`。

## 单任务循环（每条 task 走一遍）

1. **抽取**：从 `tasks.md` 取下一条**未完成**任务的全文 + 它依赖的上下文（接口、画像约定、前序任务产出的真实签名）。
2. **定级派发**：按任务难度选模型（见下），派 fresh 子代理实现，brief 含完整规格 + 反扩范围边界。子代理内部走 `implement` 的 TDD。
3. **收状态**：实现者回 DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED，按状态机处置（references）。只有 DONE（且 concerns 已解）才进评审。
4. **spec 合规评审**（独立子代理）：实现是否**逐条**满足该任务规格？缺一条 → 退回**同一实现者**修，修完重评。不过不进下一门。
5. **代码质量评审**（独立子代理）：风格/边界/错误路径/对照 `project.md` 项目特异反模式。有问题 → 退回修 → 重评。
6. **落盘 + 推进**：两门全过、`tasks.md` 勾掉该条、`TodoWrite` 同步，才回到第 1 步取下一条。

## 模型分级（机械活用便宜模型，判断活用最强）

- **机械任务**（单文件、规格完整、孤立函数、纯样板）→ 便宜/快模型。
- **集成任务**（多文件协调、跨模块、调试）→ 标准模型。
- **架构/设计判断 / 所有评审**→ 最强可用模型。评审省钱是假省——漏过的 bug 后期返工更贵。

信号：单文件 + 规格完整 = 预算档；多文件 = 标准档；需要设计判断 = 最强档。**不确定就往上调一档**，别为省钱让弱模型硬啃判断活。
全局模型路由规则见 `../flow/references/model-routing.md`：执行者可以低一档，编排/方案/评审不降档。

## 危险信号（出现即停 / 回退）

- 想"一把多派几个子代理并行跑"加速 → 停，冲突互覆，铁律 §1。
- 复用上个任务的子代理"省得重新交代上下文" → 停，串味，铁律 §2。
- 派发时只甩"看 tasks.md 第 N 条"不给全文 → 停，会读串/扩范围，铁律 §3。
- spec 还没过就先看代码质量、或两道评审让同一个子代理一起做 → 停，铁律 §4/§5。
- 子代理回 BLOCKED，你不查根因直接"再跑一次试试" → 停，铁律 §6。
- 评审子代理回"看着没问题 / should be fine / 大概 OK"却拿不出**逐条对规格**的核验 → 不算过，按未评审处理。
- 实现者自报"我测过了，过了"就采信 → 不算，要新鲜命令输出（见完成判定）。

## 正例 / 反例

- **正**：tasks.md 有 5 条。取第 1 条（"实现 parseConfig，单文件，规格含全部字段"）→ 判为机械活，派便宜模型 fresh 子代理走 TDD 回 DONE → spec 评审逐字段核对全中 → 质量评审挑出一个未处理的空输入，退回修，重评通过 → 勾掉，取第 2 条。每条独立上下文，无串味。
- **反**：把 5 条任务一次性丢给一个子代理"你都做了"，或同时开 5 个子代理并行写 → 上下文互污 / 写冲突；做完主线程自己扫一眼"看着挺对"就勾掉全部 → 无独立评审、无新鲜验证证据，作废。

## 完成判定（不由实现者自报，由 verify + Oracle 裁决）

- **单任务过门**：spec 评审 + 质量评审两份**独立子代理结论**齐全（各自写明**核对了什么** + 裁决），且实现者已贴出该任务测试的新鲜通过输出。只说"评审看过了"不算。
- **整串收口**：全部任务勾完后，**加载 `verify` 技能**按档位跑项目真实 build/test，贴新鲜 stdout + 退出码。最终完成由 `verify` + Stop hook 的 Oracle（`docs/flow/verify-cmd`）裁决，不由"我都派完了"叙述裁决。
- **封堵自报**：不信任任何子代理"跑了/过了"的口头汇报——要可核实的**命令 + 新鲜输出 + 退出码**。把规格"同义改写一遍"不算实现，"复述一遍需求"不算评审。

## checklist（声明整串完成前逐条核）

- [ ] 任务严格**串行**推进，全程没有两个实现子代理同时在写。
- [ ] 每条任务派的是 **fresh** 子代理，brief 含**完整规格全文**而非文件指针。
- [ ] 每条任务都过了 **spec→质量** 两道独立评审，顺序没颠倒。
- [ ] 评审与实现是**不同**子代理，无自评自演。
- [ ] 模型按难度分级，判断活/评审用了足够强的模型。
- [ ] 没有对 BLOCKED 做"原样重试"；每次重派都改了上下文或模型。
- [ ] 整串收口跑了 `verify`，新鲜输出已贴、退出码 0。

## 红线（反合理化）

> 违反规则字面 = 违反规则精神。任何"我遵守了精神"的解释不成立。

| 合理化借口 | 实际规则 |
|---|---|
| 一次多派几个子代理并行，快很多 | 串行，一次一个。并发写同库必冲突互覆，省的时间全赔在解冲突上（铁律 §1）。 |
| 复用上个任务的子代理省得重新交代上下文 | 每任务必 fresh。复用即串味，上个任务的"差不多"会糊进本任务（铁律 §2）。 |
| 让子代理自己去看 tasks.md 第几条就行 | 必须把任务**全文**喂进 brief。让它读文件会读串/读漏/自行扩范围（铁律 §3）。 |
| spec 大体对了，先看代码质量省一轮 | spec 没**逐条**过，不开质量评审。spec 不对，质量再高也是做错了东西（铁律 §4）。 |
| 这任务简单，我主线程顺手评审一下 | 评审必须独立子代理，实现者/主控不得自演。自评 = 无独立失败信号（铁律 §5）。 |
| 评审说"看着没问题"，那就过 | "看着/should/大概"不是评审结论。要**逐条对规格**的核验，否则按未评审处理。 |
| 子代理回报测过了，那就是过了 | 不信自报。要可核实的命令 + 新鲜 stdout + 退出码，否则等于没验。 |
| 它把需求复述了一遍，算评审过了 | 复述/同义改写不产生独立核验信号。spec 评审要逐条比对实现与规格。 |
| BLOCKED 了再跑一次也许就好了 | 绝不原样重试。先定位根因（缺上下文/模型弱/范围过大/计划缺陷）再动作（铁律 §6）。 |
| 机械活也上最强模型保险 | 机械活用便宜模型，把预算留给判断活与评审。盲目上最强是浪费而非保险（模型分级）。 |
| 全做完我扫一眼就勾完整串 | 整串收口必跑 `verify` 真实命令、贴新鲜输出，由 Oracle 裁决，不由"扫一眼"判定。 |

每条任务内部 → 用 Skill 工具加载 `implement`（builder + 独立 verifier）。整串任务全过、`verify` 收口后 → 加载 `document` 出交付物；撞上学习信号（连续 BLOCKED / 反复返工 / 评审抓到真 bug / 档位被推翻）则下一步用 Skill 工具加载 `harvest`。

遵循 `flow` 技能的质量红线。
