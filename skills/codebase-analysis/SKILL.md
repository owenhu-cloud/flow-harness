# codebase-analysis — 代码地图（只读子代理 fan-out）

陌生/大型代码库不能靠主线程逐文件 Read 硬啃：上下文会被原始代码淹没，还没看懂就溢出。本技能派**只读子代理分区扫描**，每个子代理用 Glob/Grep 侦察一个分区、只回**蒸馏后的结构化制品**，主线程汇成一张 `docs/flow/codemap.md`——刻画「这套系统内部长什么样」，供后续按图设计/施工。

**适用档位**：R2 / R3 默认；R1 遇到不熟的第三方库或遗留模块也可临时启用。
**流程位置**：在 `brainstorm`→`plan`→`implement`→`verify`→`document` 中位于 **plan 之前**——方向已大致定、但需先摸清真实结构才能设计。地图就绪后喂给 plan（基于真实结构设计）或 implement（按图施工）。

## 边界（先划清，防与邻居技能重叠）

| 这是别人的活 | 归谁 |
|---|---|
| 探测 test/build/lint 命令、固化项目画像 | `profile`（写 `docs/flow/project.md`） |
| 定代码风格 / 命名 / 反模式 | `profile` |
| 查外部网络 / 第三方文档 / 选型对比 | `research` |
| 评健康度 / 找技术债热点（churn×complexity）/ 排重构优先级 | `tech-debt-audit`（同样只读扫库，但答「哪里烂」；codemap 里别夹带债务评分，也别为审计重复全扫一遍） |
| 裁决「完成/通过」 | `verify` + Stop hook Oracle |

本技能**只回答「这套系统内部是什么样」**：结构、模块、数据流、入口、去哪找 X。不碰命令、不定风格、不上外网。需要那些 → 加载对应技能，别在这里造重复机制。

## 核心机制

1. **fan-out 分区扫描**：把代码库切成若干互不重叠的分区（按顶层目录 / 子系统 / 层），每个分区派一个**只读**子代理（`Task` 工具，建议 `Explore`）并行扫。
2. **侦察优先，不逐文件 Read**：子代理用 **Glob 列结构、Grep 找符号/调用/路由/入口**，只在信号歧义时读关键片段。禁止把整目录 Read 进上下文。
3. **回传结构化制品**：子代理只回压缩后的结构化片段（模块职责行、入口表、依赖边、证据路径），不回源码正文。
4. **主线程只做拼装**：把各分区制品缝成一张 `codemap.md`，不亲自吞原始代码。
5. **每个结论指到证据**：每条断言后跟真实 `文件路径` 或 `grep 命中`；推断不出的标 **「未确认」**，绝不臆测填空。

分区切法、子代理 brief 模板与反注入处置见 **references/zone-scan-brief.md**。

## 铁律（违反即作废）

1. **不臆测**：每条结论挂一个可追溯证据（路径 / grep 命中 / 配置行）。无证据的断言删除或标「未确认」，不靠「一般框架都这样」脑补。
2. **不逐文件 Read 污染主线程**：扫描在子代理内用 Glob/Grep 完成，主线程只接收蒸馏制品。
3. **分区不重叠、合起来全覆盖**：派发前先划清每个子代理的目录边界，避免重复扫与漏扫。
4. **制品是结构化的，不是散文**：codemap.md 用表 + Mermaid + 查找表，不写大段叙述。
5. **守边界**：命令/风格/外网一律不碰，转交 `profile` / `research`。

## 危险信号（出现即停 / 回退）

- 你正要在主线程 `Read` 整个 `src/` 看个究竟 → 停，改派只读子代理用 Grep 侦察。
- 子代理回了一坨源码原文而非结构化制品 → 退回重派，要求只回「模块职责 + 证据路径」。
- 你写下「应该是 MVC 架构」却没有任何路由/控制器文件佐证 → 删掉或标「未确认」，去 grep。
- 你开始记「测试命令是 npm test」「这个项目用 2 空格缩进」→ 越界了，那是 `profile`，停。
- 两个子代理都扫了同一个 `services/` → 分区没划清，回退重新分配。

## 产物：`docs/flow/codemap.md`

结构化「代码地图」，六块固定骨架（完整模板与填法见 **references/codemap-template.md**）：

1. **技术栈表** — 语言/框架/运行时/关键依赖 + 证据文件（manifest 路径）。
2. **架构图** — Mermaid，画主要组件与依赖方向（不臆造未见的边）。
3. **模块职责表** — 每个顶层模块：职责一句话 + 入口文件 + 主要依赖。
4. **入口点** — 进程/请求/CLI/job 的 bootstrap 位置（`文件:符号`）。
5. **一条主流程的生命周期** — 选一条代表性请求/主流程，从入口追到出口，逐跳标文件。
6. **「去哪找 X」查找表** — 意图 → 文件位置（如「加一个 API 路由 → `routes/*.ts`」）。

每块的断言都带证据列；查不实的格子填「未确认」。

## 红线（合理化借口 → 实际规则）

| 合理化借口 | 实际规则 |
|---|---|
| 这种 Spring 项目结构我熟，凭经验直接画架构图 | 凭经验即无证据。每条结论挂真实路径/grep 命中，否则标「未确认」（铁律 1）。 |
| 直接把 `src/` 整个 Read 进来看最快 | 会撑爆主上下文。扫描在只读子代理里用 Glob/Grep 做，主线程只收制品（铁律 2）。 |
| 子代理回了全文，我自己挑重点 | 子代理就是压缩器；回全文即失败，退回重派要结构化制品（机制 3）。 |
| 顺手把测试命令和缩进风格也记进 codemap | 越界。命令/风格归 `profile`，本技能只刻画内部结构（边界 / 铁律 5）。 |
| 看不出这条流程怎么走，先按常见模式补上 | 臆测污染地图。追不通就标「未确认」并指出断点，留给 plan 去查（铁律 1）。 |
| 两个子代理范围有点重叠，无所谓 | 重叠=重复扫+证据打架。派发前划清不重叠分区（铁律 3）。 |
| 地图差不多了，顺便把 plan 也写了 | 本技能只产地图。设计归 `plan`，按图施工归 `implement`，别合并角色。 |

## 正例 / 反例

- ✅ 把 monorepo 切成 `api/ web/ worker/ shared/` 四个分区，各派一个 Explore 子代理用 Grep 找路由/入口/导出，回 5–8 行结构化制品；主线程缝成 codemap.md，每行带证据路径，两处追不通的链路标「未确认（断点在 `worker/queue.ts`）」。
- ❌ 主线程 Read 了 40 个文件，凭印象写「这是个标准三层架构」，无一条路径佐证，上下文也被原始代码塞满——臆测 + 污染，作废。

## checklist（产出 codemap.md 前逐条过）

- [ ] 分区互不重叠、合并全覆盖；每个分区由独立只读子代理 fan-out 扫描。
- [ ] 主线程未逐文件 Read；扫描经 Glob/Grep 完成，只接收结构化制品。
- [ ] 六块骨架齐全：技术栈表 / 架构图 / 模块职责表 / 入口点 / 一条生命周期 / 查找表。
- [ ] 每条结论都有证据列（路径 / grep 命中 / 配置行）；查不实的标「未确认」。
- [ ] 未越界：无测试/构建命令、无风格规范、无外网内容混入。

## 交接

- 地图就绪 → **下一步用 `Skill` 工具加载 `plan` 技能**，基于真实结构做设计；若直接进施工，加载 `implement` 按图作业。
- 若尚未固化项目验证/风格规范 → 先用 `profile`，二者产物互补（codemap 讲结构，project.md 讲命令与风格）。

遵循 `flow` 技能的质量红线。
