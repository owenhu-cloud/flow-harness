# 依赖图 + 时序耦合：具体配方

本文是 `impact-analysis` 的施工手册。SKILL.md 定规则，这里给可直接照抄的命令与判据。原则不变：**每条波及关系都要落到 `file:line` 或 git 记录的证据上**，抓不全的标存疑、不静默判无影响。

## 一、反向依赖（caller — 谁调用 / 依赖入口）

目标：找出引用了入口符号或入口文件的所有位置。先按符号名搜，再按 import 路径搜，两道都要走。

通用（ripgrep，带行号 + 上下文）：
```
rg -n --no-heading '\bValidateToken\b' -g '!*_test.*'   # 符号引用，记 file:line
rg -n --no-heading 'auth/token'                          # import / 路径引用
```

按语言补抓（grep 的盲区在这里）：

| 语言/机制 | 额外要搜的 | 为什么 |
|---|---|---|
| Go | `interface` 实现：方法集匹配；`//go:generate`、wire/fx 注入 | 实现不显式 import 接口，调用走接口变量 |
| Java/Kotlin | DI 注解 `@Autowired/@Inject`、反射 `Class.forName`、SPI `META-INF/services` | 容器装配，编译期看不到调用边 |
| Python | `getattr`/`importlib`、entry_points、装饰器注册表 | 动态分发 grep 抓不到 |
| TS/JS | 动态 `import()`、字符串路由、DI token、barrel `index.ts` 再导出 | re-export 让符号名漂移 |
| 配置/IoC | YAML/JSON/env 里写的类名/路径/handler 名 | 代码里搜不到，配置注入 |

> 凡是「靠运行时才能确定调用边」的（反射/DI/接口/配置/事件总线），抓到的不全 → 把疑似消费方列进**高风险**并注「动态依赖，需人工确认」，不要因为 grep 没命中就判「无影响」。

## 二、正向依赖 + 契约变更（dependent — 改了什么会让下游崩）

不是所有改动都波及下游，但**契约类改动**会。先判本次改的是不是契约：

| 契约变更类型 | 下游受影响面 | 找下游的方法 |
|---|---|---|
| 函数签名 / 参数 / 返回类型 | 所有调用方需改解构/传参 | 同「反向依赖」搜符号 |
| 公共类型 / struct 字段 / schema | 序列化、持久化、跨服务消费方 | 搜类型名 + 字段名 + DB 迁移 |
| 事件 / 消息 / topic 名 | 订阅方（可能跨仓库/服务） | 搜 topic 常量、订阅注册 |
| HTTP/gRPC/API 契约 | 客户端、前端、SDK | 搜 route、proto、OpenAPI 引用 |
| 错误码 / 枚举 / 常量语义 | 依赖具体值分支的代码 | 搜枚举名 + 具体值 |
| 配置键 / 默认值 | 读该配置的所有点 + 运维 | 搜配置键字符串 |

纯内部实现改动（不动签名/类型/可观测行为）→ 下游通常**不**受影响，但仍要在「已排除」里写明「仅内部实现，契约未变」。

## 三、二跳传递

对第一步得到的每个直接受影响文件，再做一次反向 caller，得到间接受影响。标传递路径，例如 `token.go → middleware.go → router.go`。

止步规则（避免无限扩散）：
- 到达**稳定接口 / 已发布 API / 外部边界**即停（其契约视为不变）。
- 到达有充分测试覆盖、且本次不改其契约的层即停。
- 一般 2–3 跳足够；超过说明改动触及核心抽象，整体标高风险。

## 四、git 时序耦合（temporal coupling）

历史上「总是一起改」的文件，往往有 grep 看不见的隐性耦合（共享假设、约定、数据格式）。

入口文件的改动史与共改文件：
```
git log --follow --format='%H' -- path/to/token.go        # 入口的提交集
# 对每个提交列出同提交改的其它文件，统计频次：
git log --follow --name-only --format='---' -- path/to/token.go
```
一行式统计共改频次（按出现次数排序）：
```
git log --follow --pretty=format: --name-only -- path/to/token.go \
  | sort | uniq -c | sort -rn | head -20
```

判读：
- 共改次数高 **且** 第一/二步已发现代码依赖 → 强证据，进直接/间接受影响。
- 共改次数高 **但** 无代码引用 → 单列「时序耦合（无代码依赖证据）」，**交人判断**，提示可能是测试、配置、约定耦合。不强行删、也不强行当依赖。
- 共改次数低（1–2 次）→ 噪音，忽略或只在「已排除」记一笔。

## 五、受影响集 → 测试集映射

把受影响文件映射到覆盖它们的测试：
- 同目录/同包测试（`*_test.go`、`test_*.py`、`*.spec.ts`）。
- 按被测符号反查测试（搜测试里出现的入口符号名）。
- CI 配置里这些路径触发的 job（读 `docs/flow/project.md` 固化的命令，别现场瞎猜）。

产出的测试集是 **verify 的下限**：verify 必须按它跑足、不取子集。若某受影响文件**无任何测试覆盖** → 这是高风险信号（见下）。

## 六、高风险打分（触发强制 code-review）

任一命中即标高风险，并在 impact.md 注明理由：

| 风险因子 | 判据 |
|---|---|
| 高扇入 | caller 数量多（如 ≥ 5 处直接调用） |
| 安全敏感 | 鉴权 / 鉴别 / 加密 / 权限 / token / 密钥 |
| 资金/合规敏感 | 支付 / 余额 / 计费 / 账务 / 审计 |
| 并发/状态 | 锁、共享状态、事务、幂等、消息顺序 |
| 无测试覆盖 | 受影响文件没有对应测试 |
| 动态依赖存疑 | grep 抓不全的反射/DI/接口/配置消费方 |
| 深传递 | 二跳仍未收敛、触及核心抽象 |

高风险点 → 在 implement/verify 阶段提示用 `Skill` 加载 `code-review` 强制复核；其余照常走 verify。
