# impact-analysis — 改之前先算波及面

适用档位 **R2 / R3**（任务驱动，非常驻）。在 Flow 流程中的位置：`brainstorm（门）→ 【impact-analysis】→ plan（门）→ implement → verify → document`。
在动手改某个符号/文件**之前**跑一次：算清「这次改动会动到谁」，输出**受影响文件集 + 应跑测试集 + 高风险点**，把范围交给 `plan`（排任务、定测试范围）和 `verify`（按受影响集跑足）。

唯一产物是文档 `docs/flow/<change>/impact.md`，**不写代码、不跑实现**。

## 边界（越界即换技能，别在本技能里做）

- 不探测/固化构建测试命令 → 那是 `profile`（画像写在 `docs/flow/project.md`，本技能直接读用，不重复探测）。
- 不查外网 / 不做技术选型调研 → 那是 `research`。
- 不画整库架构地图、不梳理全模块依赖 → 那是 codebase-analysis 类任务。
- 本技能**任务驱动、聚焦单次改动的半径**：只回答「改这处会波及谁」。

## 核心机制（证据驱动，每条波及关系都指到证据）

对将改的符号/文件，构建依赖图 + 叠加 git 时序耦合，六步：

1. **锚定入口**：列出本次将改的符号 / 文件，作为分析锚点（与 plan 的 change 同名）。
2. **反向 · caller**：grep 符号名 / import 路径，找**谁调用 / 谁依赖**入口 → 直接受影响。每条记 `file:line`。
3. **正向 · dependent + 契约**：看入口依赖谁；若改的是**签名 / 返回类型 / 事件 / schema / 公共契约**，找下游消费方 → 也是直接受影响。
4. **二跳传递**：对直接受影响再做一跳（caller 的 caller），标传递路径，止于稳定边界（如稳定接口 / 外部 API）。
5. **时序耦合**：`git log` 入口文件，统计**历史上常与它一起改**的文件 + 共改次数 → 补充证据集。
6. **汇总**：受影响集 → 测试集映射（哪些测试覆盖了这些文件）；标出高风险文件（高扇入 / 鉴权 / 支付 / 并发 / 无测试覆盖）。

grep/git 具体配方（按语言）、契约变更分类、高风险打分见 `references/dependency-graph.md`（动手前先读）。

## 铁律（iron-laws，违反即作废）

1. **无证据不进清单**。每条波及关系必须指到 `file:line` 的 import / 调用，或 git 共改记录。指不出 = 臆测 = 删。
2. **双向都算**。caller（谁调它）+ dependent（它调谁 / 契约变了谁在下游消费）。只看一向必漏一半。
3. **时序耦合是信号不是依赖**。git 共改高但无代码引用证据的，单列「时序耦合」类、注明「无代码证据」，交人判断，**不强行当依赖删也不强行算依赖**。
4. **受影响测试集是 verify 的下限，不是上限**。verify 按此集跑足、不取子集；本技能不得为「省事」预先缩小它。
5. **grep 抓不全的要标存疑**，不静默判无影响：动态调用 / 反射 / 接口实现 / 配置注入 / DI 容器，抓不到的疑似点进高风险，提示人工确认。
6. **完成判定仍走 verify + Stop hook Oracle**（`docs/flow/verify-cmd`）。impact.md 只定范围，不替代验证，不自报「改这处安全」。

## 合理化借口 → 实际规则（压力下逐条钉死）

| 你会对自己说的话 | 实际规则 |
|---|---|
| 「这函数我熟，波及面我心里有数」 | 心里有数 ≠ 证据；每条波及关系指到 `file:line` 的 import/调用，指不出就删 |
| 「跑全量测试最保险，不用算了」 | 大库全量不可行且掩盖真实耦合；算受影响集是为了「跑足且不漏」，不是为了少跑 |
| 「只看谁调用它就够了」 | 双向：caller + dependent + 契约下游；只看一向必漏 |
| 「grep 没引用，应该没人用了」 | 动态调用/反射/接口实现/配置注入 grep 抓不全；存疑标高风险，不静默判无影响 |
| 「git 共改它俩，肯定有依赖，写进直接受影响」 | 时序耦合是信号不是依赖；单列「时序耦合」类、注明无代码证据，交人判断 |
| 「波及面太大，挑几个重点写」 | 漏掉的就是 verify 跑不到的回归盲区；要么全列，要么对每个排除项显式记「已确认无关 + 理由」 |
| 「顺手把整个模块架构也梳理一遍」 | 越界到 codebase-analysis；本技能只答这次改动的半径 |
| 「测试命令我现场探一下」 | 先读 `docs/flow/project.md` 的固化命令；缺失才提示先跑 `profile` |
| 「算完直接开始改」 | impact 是输入不是终点；范围喂 `plan`/`verify`，高风险喂 `code-review` |

## 危险信号（出现即停 / 回退）

- 准备写「应该没别的地方用了」，却没真跑过 grep / 调用查找。
- 受影响清单里有条目说不出证据来源（哪个 file:line / 哪条 git 记录）。
- 只列了 caller，漏了被改契约的下游 / 接口实现方。
- 想跳过第 5 步 git 时序耦合（「太麻烦」）。
- 算出的测试集比直觉小很多，却没复核是不是漏了动态依赖。
- 开始描述整库分层 / 画大架构图 —— 越界信号。

## 正反例

**反例（违规）**：「改 `auth.ValidateToken`，影响不大，跑下 auth 包测试就行。」
→ 无依赖图、无调用证据、凭感觉缩范围、漏下游契约消费方。失格。

**正例（合规）**：「改 `auth.ValidateToken` 返回类型。
反向 grep 调用方：`api/middleware.go:42`、`api/grpc.go:88`（直接受影响）；`middleware.go` 被 `router.go:31` 装配（间接，二跳）。
契约：返回类型变 → 下游 `api/handler.go:120` 解构受影响。
时序耦合：`git log --follow auth/token.go` 近 10 次改动，7 次连带改 `auth/token_test.go`、5 次连带 `api/middleware_test.go`（标：含代码依赖，强证据）。
受影响测试集 = `./auth ./api`（喂 verify，跑足不取子集）。
高风险：`middleware.go` 扇入 3 处且涉鉴权 → 标 code-review。」
→ 双向、每条带证据、测试集明确、高风险点交接。

## 产出（给 AI，密集可解析）

`docs/flow/<change>/impact.md`，固定骨架：

```markdown
# <change> 波及面
## 变更入口        —— 将改的符号/文件清单（分析锚点）
## 直接受影响      —— caller + dependent + 契约下游；每条带 file:line 证据
## 间接受影响      —— 二跳传递，标传递路径，止于稳定边界
## 时序耦合        —— git 常共改文件 + 共改次数；逐条标「含/无 代码依赖证据」
## 应跑测试集      —— 覆盖受影响集的测试文件/命令（= verify 下限，喂 verify）
## 高风险文件      —— 触发强制 code-review 的点 + 理由（扇入/鉴权/支付/并发/无测试覆盖）
## 已排除          —— 看似相关但确认无关的项 + 排除理由（防「漏了」与「臆测」两头）
```

一份填满的完整范例见 **`references/impact-example.md`**——照其密度与证据强度产出。

## 进门后 checklist（逐条勾，缺一条不交接）

- [ ] 已读 `docs/flow/project.md` 取真实测试/构建命令（缺失则提示先跑 `profile`）
- [ ] 入口符号/文件列全，与 plan 的 change 同名
- [ ] 反向 caller 与 正向 dependent + 契约下游**都查过**（双向）
- [ ] 每条受影响关系指得到 `file:line` 或 git 证据，无臆测
- [ ] 动态调用/反射/接口实现/DI 的存疑点已标进高风险，未静默判无影响
- [ ] 跑过 git 时序耦合，逐条区分了「含/无代码依赖」
- [ ] 应跑测试集写明，且声明它是 verify 的下限（不可取子集）
- [ ] 高风险文件列出 + 理由，需要时提示走 `code-review`
- [ ] 没写代码、没跑实现；产物只有 `impact.md`

## 收尾 → 交接

波及面算清后：
- 受影响集 + 测试集 → 用 `Skill` 工具加载 `plan`（据此排任务、定测试范围）；plan 收口后由 `verify` 按受影响集跑足。
- 若标出高风险文件 → 在 implement / verify 阶段提示用 `Skill` 工具加载 `code-review` 对这些点强制复核。
- 完成判定由 **verify + Stop hook Oracle** 裁决，不由本技能自报「安全」。

遵循 `flow` 技能的质量红线。
