# document — 人类交付物（R1+）

只产**给人看**的产物：读者要的是「现在是什么、为什么这么选、怎么用」，不是 agent「先…然后…接着…」的施工日记。默认中文（随项目主语言）。

## 适用档位与流程位置

- 档位：R1+。R0（一次性小改）不写文档。R1 通常一段 PR body 或一节 CHANGELOG 即够；R2/R3 才写 `docs/<feature>.md` 设计文档。
- 位置：`brainstorm → plan → implement → verify → document`。**document 在 verify 之后**——文档描述的是已被验证命令裁决为真的状态，不是计划或愿望。verify 未过就不该有「已完成」的交付文档。

## 产出（按档位裁剪，宁缺毋滥）

- `docs/<feature>.md`：结论先行（这是什么 / 关键决策 / 怎么用），后接设计取舍与一两张关键图（图调 `diagram` 技能产 Mermaid）。
- PR body：变更意图 + 影响面 + 验证方式（引用 verify 的新鲜输出，不复制粘贴全部日志）。
- CHANGELOG：面向使用者的行为变化，一行一条。

## 铁律（iron-laws，违反即作废）

1. **交付物 ≠ 过程日志**。读者是人类维护者/用户，不是回放 agent 的执行轨迹。写结论与取舍，删施工流水账（红线 §5）。
2. **只写已成真的状态**。文档描述当前代码的事实，不写「打算」「将会」「TODO 之后补」。未做的事不进交付物，进 issue。
3. **取舍要写「为什么不选 B」**。只夸方案 A 不算设计取舍；写下被否决的备选和否决理由，文档才有维护价值。
4. **声明的行为必须可被 verify 复核**。文档里「已支持 X」必须对应 verify 已跑绿的命令；没有证据支撑的「已完成」不许写。

## 危险信号（出现即停，回去重写）

- 正文出现「我先运行…接着发现…然后修改了…最后…」——这是日志，不是文档。
- 段落在复述 git 提交顺序或调试历程。
- 通篇只有「做了什么」，没有「为什么这么做 / 还能怎么做但没选」。
- 出现 should / seems / 大概 / 应该没问题 / 理论上——结论文档里禁用模糊措辞，要么是事实要么删掉。
- 文档篇幅随迭代次数膨胀（每轮都往里追加一段心路）——交付物只反映终态，不累加历史。

## 红线（反合理化）

| 合理化借口 | 实际规则 |
|---|---|
| 把调试过程写进去能体现工作量 | 工作量不是交付价值。读者要终态与取舍，过程日志一律删（铁律 1）。 |
| 先写「将支持/计划支持」占个位 | 文档只写已成真且 verify 可复核的状态；未做的进 issue/TODO，不进交付物（铁律 2）。 |
| 方案讲清楚就行，备选不用提 | 不写「为何不选 B」就不算设计取舍，文档失去维护价值（铁律 3）。 |
| verify 还没跑完，先把文档写成已完成 | document 在 verify 之后；没有绿色证据不许写「已完成」（铁律 4 / 流程位置）。 |
| 这轮改动小，往老文档末尾追加一段就好 | 交付物反映终态，需就地更新而非堆叠心路，否则文档越长越没用（危险信号）。 |
| 措辞写「应该能正常工作」稳妥些 | 结论文档禁用 should/seems/大概；是事实就断言并给证据，不确定就别写。 |
| 同义改写一遍代码注释就算文档了 | 复述实现细节不是给人看的交付物；要给出代码读不出的决策理由与权衡。 |

## 正反例

**反例**（流水账，作废）：
> 我先加了 `cache.go`，跑测试报错，于是改了锁的顺序，又发现并发问题，最后用 sync.Map 解决，应该没问题了。

**正例**（交付物）：
> ## 缓存层
> **结论**：读多写少，采用进程内缓存，TTL 5 min。
> **取舍**：未选 Redis——当前单实例部署，额外运维成本不划算；用 `sync.Map` 而非 `map+RWMutex`，因热点 key 集中、读路径无锁更优。
> **代价**：多实例水平扩展时需改为 Redis，已记入 #142。

## Checklist（交付前自检）

- [ ] 第一屏就能读到「这是什么 / 关键决策 / 怎么用」（结论先行）。
- [ ] 每个关键决策都写了被否决的备选及理由。
- [ ] 没有任何「我先/然后/接着」式过程叙述。
- [ ] 没有 should/seems/大概/将会 等模糊或未来时措辞。
- [ ] 所有「已完成/已支持」都能对应 verify 的新鲜证据。
- [ ] 需要图时已用 `diagram` 技能产出 Mermaid 并嵌入。
- [ ] 中英文排版规范（见 `references/typography.md`）。

中文排版与术语细则（空格、标点、术语保留）见 `references/typography.md`。

## 交接

文档落定即流程终点。若文档写作中发现实现与设计不符或有未验证声明 → 回退 verify/implement，别在文档里粉饰；本轮 change 若撞上学习信号（返工、档位被推翻等）→ 下一步用 Skill 工具加载 `harvest` 技能沉淀经验。

遵循 `flow` 技能的质量红线。
