# writing-skills — 写技能（对流程文档套 TDD）

写 skill = 把 TDD 套到流程文档上。先有失败，再有规则。

## 技能就是一个 Markdown 文件（无 CLI、无注册表）

新增一支技能 = 建 `skills/<name>/SKILL.md`，Claude Code **自动发现**，无需任何注册命令。最小格式：

```yaml
---
name: <kebab>           # 必须等于目录名
description: <触发条件>  # "...时用"；只写何时触发，别把工作流步骤写进来
---
```

- `description` 决定何时加载本技能——**只写触发条件 + 可搜索的症状/同义词**，前置用户会怎么描述这个问题、会冒什么报错关键词（CSO）。
- **禁**把正文步骤塞进 description，否则 agent 把它当捷径、跳过正文。
- 正文按需加载，保持密集；长参考（>100 行）拆到 `skills/<name>/references/`，正文留指针。

## 铁律：无失败场景，不写 skill

下笔前先写出**具体失败场景**：一个**没有此 skill** 的 agent 会在哪一步做错、并如何自我合理化。
写不出该场景 → 不存在需求 → 不写。场景越具体（含 agent 原话级借口），后续规则越精准。

## 循环：红 → 绿 → 重构

| 阶段 | 动作 |
|---|---|
| 红(Red) | 记录无 skill 时 agent 的具体行为与合理化原话（「这是小改动，跳过测试」之类）。 |
| 绿(Green) | 写**最小** skill，逐条精确针对上述借口，不多写。 |
| 重构(Refactor) | 找 agent 在新规则下发明的新漏洞 / 新借口，补一条反驳。重复直到无懈可击。 |

每轮只补真实出现过的漏洞，不预防想象中的问题。

## 行为塑造类技能必含「红线表」

凡是 agent 容易在压力下合理化绕过的纪律（反 reward-hacking 等），正文必须含一张「合理化 → 规则」对照表，把每条已观察到的借口钉死：

| 合理化 | 规则 |
|---|---|
| 「测试太慢，先跳过」 | 提交前必跑真实验证命令，无例外。 |

## 接入

把新技能正文末行接上：`遵循 `flow` 技能的质量红线。` 让它继承统一纪律。建好文件即生效——顺手在 `flow` 技能的「技能地图」补一行，方便路由时被想起。

遵循 `flow` 技能的质量红线。
