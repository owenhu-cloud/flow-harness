# writing-skills — 写技能（对流程文档套 TDD）

写一支技能 = 把 TDD 套到流程文档上：**先有失败场景，再有规则**。技能不是说明书，是用来纠正"未来某个 agent 会犯的具体错误 + 它会怎么自我开脱"的纠偏器。

## 适用档位与流程位置

这是**元技能**：它本身不分 R0–R3，而是在你要新增/改写任何一支 Flow 技能（R0–R3 全套）时启用。它不属于 brainstorm→plan→implement→verify→document 主线，而是**生产这些技能的工具**——但产出物必须能嵌进主线、并继承 Flow 身份（见下「身份不可丢」）。

## 铁律（iron-laws，违反即作废）

1. **无失败场景，不写技能。** 下笔前必须先写出一个具体场景：一个**没有这支技能**的 agent 在哪一步做错、会说出哪句原话级借口。写不出 → 不存在需求 → 不写。
2. **frontmatter 只有 `name` 与 `description`。** `name` 一字不差等于目录名；`description` 只写"何时触发 / 可检索症状"，末尾接「 · EN: <英文 CSO 关键词>」。**禁**把正文步骤塞进 description。
3. **description 不是摘要。** 它决定"何时被加载"，不是"加载后做什么"。塞进工作流步骤 → agent 把它当捷径、跳过正文。
4. **行为塑造类技能必含红线表。** 凡涉及纪律、测试、验证、完成判定，正文必带「合理化借口 → 实际规则」对照表（见下）。
5. **身份不可丢。** 每支技能正文须：标明适用档位与流程位置 → 末尾点明「下一步用 Skill 工具加载 X 技能」的显式交接 → 末行一字不差是 `遵循 `flow` 技能的质量红线。`。少一项即作废。
6. **薄而密。** 单个深主题或正文超 ~120 行 → 拆到 `references/<topic>.md`，正文留一句指针。主文件不堆背景、不讲故事。

## 红 → 绿 → 重构

| 阶段 | 动作 |
|---|---|
| 红(Red) | 记录无此技能时 agent 的真实行为与**原话级**合理化（「这是小改动，跳过测试」「子代理说过了，应该没问题」）。纪律类至少凑 3 种压力情境。 |
| 绿(Green) | 写**最小**技能，逐条精确反驳上述借口，不多写一条。 |
| 重构(Refactor) | 找 agent 在新规则下发明的**新**漏洞/新借口，补一条钉死。重复到无懈可击。 |

只补**真实出现过**的漏洞，不预防想象中的问题。技能膨胀 = 没做重构。

## 行为塑造类必含「红线表」

把每条已观察到的开脱借口逐字钉死。反 reward-hacking 是重点：凡涉及测试/验证/完成判定，必须封堵下列借口。

| 合理化借口（agent 原话） | 实际规则 |
|---|---|
| 「这技能太简单，不用先写失败场景」 | 写不出失败场景就是没需求，删掉别写。 |
| 「我把别的技能改写一遍当新技能」 | 同义改写不算新技能；没有新的失败场景就是重复。 |
| 「description 里顺便写清步骤更方便」 | description 只写触发条件；步骤进正文，违者 agent 跳读。 |
| 「这条规则我觉得 agent 应该懂」 | "应该/大概/seems" 不是规则；写出可检验的硬约束或不写。 |
| 「验证太慢，技能里写'手测即可'」 | 完成判定一律交 verify + Stop hook Oracle 裁决，不得在技能里开后门。 |
| 「子代理自报通过，我就写'信任结果'」 | 自报不算证据；技能须要求贴新鲜命令输出，不得信任自报。 |

## 危险信号（出现即停，回退重写）

- description 里出现动词流水（"先…再…然后…"）→ 步骤泄漏，搬进正文。
- 技能没有任何"出现 X 即停"的硬规则，全是建议口吻 → 不是纪律，是散文。
- 正反例缺失，或例子是抽象占位（"做正确的事"）→ 补具体到能照抄/照避的例子。
- 正文 > 120 行还在堆背景 → 拆 references。
- 末行不是那句质量红线，或缺交接/档位 → 身份残缺，作废。

## 正反例

**反例（description 泄漏步骤）：**
`description: 修 bug 时用——先复现，再定位，再写测试，再改，再回归。`
→ agent 读完 description 自以为懂了，跳过正文里的对抗式验证。

**正例（只写触发症状 + CSO）：**
`description: 声明"完成/通过"前用——手测、口头"应该没问题"不算。 · EN: before claiming done — run real build/test, paste fresh output.`
→ 命中触发后 agent 仍须进正文取规则。

## 落地 checklist（逐条勾，缺一不发布）

- [ ] 先写下了具体失败场景（含 agent 原话级借口）。
- [ ] frontmatter 仅 `name`(==目录名) + `description`(触发条件 + ` · EN:` 关键词)。
- [ ] 正文标明适用档位 + 在 brainstorm→…→document 中的位置。
- [ ] 行为/纪律类含红线表；测试/验证类封堵了同义改写、信任自报、should/seems 三类借口。
- [ ] 至少 1 组具体正反例 + 一个"出现即停"危险信号。
- [ ] 深主题/超 120 行已拆到 `references/`，正文留指针。
- [ ] 末尾有显式交接「下一步用 Skill 工具加载 X」。
- [ ] 末行一字不差：`遵循 `flow` 技能的质量红线。`
- [ ] 不发明与现有机制冲突的新机制（完成判定→verify+Oracle；项目画像→docs/flow/project.md；升维表→flow 技能）。

## 原生最小格式与渐进披露

技能就是一个 Markdown 文件——建 `skills/<name>/SKILL.md`，Claude Code **自动发现**，无需任何注册命令。最小格式、CSO 写法、references 拆分阈值与渐进披露机制，详见 `references/native-format.md`。

## 交接

技能写完后：自检走完上面的 checklist → **下一步用 Skill 工具加载 `verify` 技能**，把"新技能在真实任务里是否真的纠正了目标失败"当作待验证命题，别口头宣称生效。若发现技能本身引出新失败信号，回到 `harvest` 沉淀根因。

遵循 `flow` 技能的质量红线。
