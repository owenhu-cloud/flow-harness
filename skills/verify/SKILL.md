# verify — 完成必带证据

适用全档位（R0–R3），位于 brainstorm→plan→implement→**verify**→document 的收口处。**「完成」由真实验证命令的新鲜输出裁决，不由 agent 自说自话。** 这一支技能塞死 reward-hacking：你只能用刚跑出来的证据，不能用措辞、记忆或别人的口供。

## 铁律（iron-laws，违反即未完成）

1. **无新鲜证据不许声明完成。** 「新鲜」= 本轮、本次代码状态下，刚跑完整命令的真实输出。
2. **声明↔证据一一对应。** 每一句「X 通过了」都必须能指向一段贴出的命令输出。指不出 = 没通过。
3. **不取子集、不下调档位。** 跑该档位对应的完整命令集；绿一半不等于绿。
4. **不信任何转述。** 子代理自报「已通过」、上一轮的旧绿、CI 上次的结果，都不是本轮证据。
5. **同义改写也算违规。** 把「完成」换成「搞定了 / 应该 OK / 基本没问题 / 跑通了」绕过验证，等同伪造。

## 验证闸门（5 步，跳一步即失格）

1. **指认**：哪条命令能证明这个结论？
2. **执行**：完整跑一遍（fresh，全量，不截断）。
3. **读取**：看完整输出，核退出码，数失败条数（不是扫一眼就关）。
4. **校验**：输出是否真的支持该声明？失败数必须为 0。
5. **才能**：说出结论，并把证据贴上。

## 找到项目真实的验证命令（可靠性递降）

0. **项目画像**：先看 `docs/flow/project.md`。存在即用其中固化的 test/build/lint 命令，不重复探测（由 `profile` 维护）。不存在则按下列阶梯现场探测，并提示首个任务应先跑 `profile` 固化。若 `flow-bootstrap` 提示画像过期（清单晚于 `project.md`），先回 `profile` 更新再用其命令，别拿过期命令验证。
1. **CI 配置**（`.github/workflows/*`、`.gitlab-ci.yml`、`.circleci/` 等）——最可靠：CI 真正跑的就是验证函数，照搬它的命令。
2. **包/构建清单**（`package.json` scripts、`Makefile`、`Cargo.toml`、`pyproject.toml`、`go.mod`）。
3. **代码采样**：测试目录与命名约定、测试框架。

降级链：有 CI 用 CI；无 CI 用清单推断；无测试则退化为 `build + lint`；连构建都没有 → 明确告诉用户「该任务无法自动验证」，退回人工确认，**不静默判通过**。

## 维护 Oracle 的燃料

确定验证命令后，把能独立裁决「完成」的那条（测试 / 构建命令）写进 `docs/flow/verify-cmd`（单行）。这是 `Stop` hook Oracle（`hooks/flow-oracle.sh`）的燃料：写下它，agent 此后每次收尾都会被独立进程复核，绕不过去。命令变化时同步更新；无可自动验证命令时留空。完成判定由 **verify + Stop hook Oracle** 共同裁决——你绕过 verify，Oracle 仍会拦你。

## 按档位跑足深度

| 档位 | 底线 |
|---|---|
| R0 | 冒烟（build / smoke） |
| R1 | 单测 |
| R2 | 单测 + 集成 |
| R3 | + 关键路径 E2E |

跑该档位对应的**完整命令集**，不取子集、不下调档位。

## 声明 ↔ 证据 映射（左列声明，必须有中列证据；右列不算数）

| 声明 | 必需证据 | 不充分（即视为未验证） |
|---|---|---|
| 测试通过 | 测试命令输出：failures=0 + 退出码 0 | 上一轮跑过、「应该过」、只跑了部分 |
| 无 lint 错误 | linter 输出：0 error | 「改完应该没问题」、推断 |
| 构建成功 | 构建命令：exit 0 | lint 过了、日志看着没报错 |
| bug 已修 | 复现原症状的用例现在通过 | 代码改了、假设它好了 |
| 回归用例有效 | 红→绿循环都看过（先红后绿） | 只看到过一次绿 |
| 子代理已完成 | VCS diff / 实际文件改动可见 | 子代理回报「已完成」 |
| 需求满足 | 逐条对照验收清单核过 | 测试过了就当需求满足 |

## 危险信号（出现即停，回退到「执行命令」）

- 准备打出 **should / seems / probably / 应该 / 大概 / 基本上 / 八成 / 跑通了** 这类词来描述结果。
- 在跑命令**之前**就表达满意、想收尾、想 commit/push。
- 想「这次就破例一下」「上轮跑过这轮没大改」「环境没配好先跳过」。
- 引用**子代理的自报结论**当证据，而没自己看 diff/输出。
- 只跑了完整命令集的一部分就想下结论。

## 正反例

**反例（违规）**：「我把空指针修了，逻辑很直接，单测应该能过，标记完成。」→ 没有新鲜输出、用了「应该」、未跑命令。直接失格。

**正例（合规）**：「修复后跑 `pytest tests/ -q`：`94 passed, 0 failed`（exit 0），并补了复现该 NPE 的 `test_null_user` 现在 passed。R1 档位单测命令集全绿，证据如下 …」→ 命令 + 退出码 + 失败数 + 针对性回归，贴出。

## 产出

验证报告：贴**新鲜的**工具输出（命令 + 退出码 + 关键日志 + 失败条数）。失败如实记录，不粉饰、不挑好看的贴。

## 红线（反合理化）

| 合理化借口 | 实际规则 |
|---|---|
| 本地手测过了就行 | 手测不是证据。完成声明须附该档位自动化命令的新鲜输出。 |
| 这个档位跑单测就够了不用集成 | 底线按档位表分级：R2 含集成，R3 含 E2E，不得下调。 |
| 上一轮跑过了这轮没大改不用再跑 | 「完成/通过」声明须在同一轮、当前代码状态下附新鲜输出，不复用旧结果。 |
| 测试环境没配好先跳过这步 | 环境缺失即该档位未验证，回人工确认，不静默判通过。 |
| 跑了一部分绿了就声明通过 | 须运行档位对应的完整命令集，不取子集。 |
| 子代理说跑过了，我信它 | 不信任何转述；自己看 diff / 输出再下结论。 |
| 不说「完成」，说「搞定了/应该 OK」就不算声明 | 同义改写也是声明，照样要证据，否则等同伪造。 |
| 我很有把握，没必要再跑 | 把握 ≠ 证据；唯一能消除「未验证」状态的是跑命令。 |

## checklist（声明完成前逐条打钩）

- [ ] 用了 `docs/flow/project.md` 或阶梯探测得到的**项目真实**命令，不是通用猜测。
- [ ] 跑了**本档位完整命令集**（R2 含集成 / R3 含 E2E）。
- [ ] 输出是**本轮、当前代码**刚跑出来的，failures=0、退出码 0。
- [ ] 每句「通过」都指得到一段贴出的证据。
- [ ] 子代理结论自己复核过 diff/输出，没有直接采信转述。
- [ ] 报告里没有 should/seems/应该/大概 这类规避措辞。
- [ ] `docs/flow/verify-cmd` 已同步当前裁决命令（喂给 Oracle）。

高风险/高不可逆面想用**异模型**对抗证伪增强独立性（不替代本技能跑命令）→ 用 Skill 工具加载 `cross-verify`。

下一步：验证全绿且需要给人看的交付物时，用 Skill 工具加载 `document` 技能产出结论；若 verify 抓到真 bug 或反复返工，用 Skill 工具加载 `harvest` 沉淀经验。

遵循 `flow` 技能的质量红线。
