# false-positive-gate — 误报二次复核门

借鉴「强制 gate 复核降误报」：每条发现在进报告主表前，过一道反问门。浅层模式匹配的最大失败模式是**把正常代码当债**——这道门专治它。漏报不致命（下次复跑会补），**误报会污染整张优先级表、误导 plan 排错任务**，所以宁可严。

## flag 前强制反问清单（逐条过，任一不过就降级或转入「看着差其实合理」）

对每个候选发现问：

1. **有 file:line 吗？** 没有坐标 → 不是发现，是印象，删。（铁律1）
2. **churn 支持吗？** 复杂/巨大但近半年 churn≈0 → 不是债，是稳定资产 → 转「看着差其实合理」。
3. **它真的常出问题吗？** 翻 `git log -- <file>` 与 commit message：这文件是否反复因 bug 被改？有 bug 史 → 加权；零事故 → 降级。
4. **这写法是无意的烂，还是有意的取舍？** 读注释/commit 找意图。刻意的性能 hack、兼容垫片、防御冗余 → 标注意图，不当债。意图不明 → 转待核，不进高危。
5. **抽象真的更好吗？**（针对重复）三处是否真同构、会否一起变？偶然雷同强抽象会造耦合债 → 标注权衡，不拍板「必须 DRY」。
6. **严重度能从信号推出来吗？** 说不出「为什么高危」的客观理由（churn/复杂度/覆盖率/bug 史）→ 你在凭手感，降级。

过不了 2/3/4/5 的，大多数应落到报告的**「看着差其实合理」节**，而不是消失——记录「考虑过、为何不flag」本身就是审计价值，也防止复跑时反复纠结同一处。

## 各类「看着差其实合理」判定细则

| 表象 | 何时确实是债 | 何时其实合理（不flag / 入反节） |
|------|--------------|--------------------------------|
| god-file（行数超阈值） | 高 churn + 多职责 + 改动易扩散 | 低 churn + 单一内聚职责（解析器/状态机/生成代码） |
| 零测试覆盖 | 高 churn + 高复杂 + 有 bug 史 | 低 churn + 低复杂 + 纯数据/常量/样板 |
| 高圈复杂度 | 常改 + 无测试 + 嵌套源于堆补丁 | 本质复杂度（算法/协议解析）+ 有测试 + 稳定 |
| 重复 3+ 次 | 真同构 + 历史上一起改 | 偶然雷同 + 各自独立演化（强抽象反造耦合） |
| 反常/丑写法 | 无意为之、无注释、反复踩坑 | 有意的性能/兼容 hack，注释或 commit 说明了意图 |
| 大量 TODO/FIXME | 集中在高 churn 热点、阻塞改动 | 散落的低优先备忘，不在活跃路径 |
| 深继承/复杂依赖 | 改一处牵连多模块 + 常改 | 框架要求的固定结构、稳定不动 |

## 复跑（活文档）时的 gate

复跑不是重写，是**逐条复核 + 增量**：

- 上次的每条：去现场看还在不在、是否已修。修了 → 标 `RESOLVED(日期)`，保留行做历史。仍在 → 重新核 churn（可能升降级）。
- 重挖 churn 后冒出的新热点 → 标 `NEW`。
- 不许「上次写过就直接抄过来」——churn 是动的，上次高危这次可能已冷却（没人再改了），该降级。

复跑后在「复跑摘要」节汇总 RESOLVED / NEW / 仍存在三类计数，让人一眼看出债务是在还还是在涨。
