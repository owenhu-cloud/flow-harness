# tech-debt-audit — 技术债审计（热点定位 + 活文档）

泛泛的 code review 给的是「这段可以更好」；本技能给的是**这个库哪里最烂、按什么顺序还、动哪块最危险**——用客观信号（改动频率 × 复杂度）定位热点，每条发现钉到 `file:line`，落成可复跑的审计报告。

**适用档位 R2 / R3。**（R0/R1 单点小改不必体检。）流程位置：在 `brainstorm → plan → implement → verify → document` 的**最前面、与 profile 并列的诊断阶段**——给「要不要重构、先重构哪里」提供事实底座，喂给下游 `plan` 排任务。它产**诊断结论**，不改一行代码。

开工前先读 `docs/flow/project.md`（profile 的项目画像）：技术栈、构建/测试命令、项目特异反模式直接复用，不重探。无画像 → 先用 `profile` 固化。

## 与 profile 的边界（别造重复机制）

| | profile | tech-debt-audit |
|---|---|---|
| 问题 | 这库**怎么跑、什么风格**？ | 这库**哪里烂、动哪最危险**？ |
| 产物 | `docs/flow/project.md`（画像，给 agent 读） | 根目录 `TECH_DEBT_AUDIT.md`（债务报告，给人读+复跑） |
| 关系 | 本技能的**输入** | 复用 profile 的栈/命令/反模式 |

profile 探「环境与规范」，本技能评「健康与风险」。不重复探栈、不重复跑 lint 当审计——读画像即可。

与 `codebase-analysis` 也别混：codebase 答「内部结构长什么样」产 `codemap.md`，本技能答「哪里烂/动哪危险」产 `TECH_DEBT_AUDIT.md`。两者都只读 fan-out 扫库——**已有 codemap 就复用它的模块边界做分区，别为审计把整库重复扫一遍**。

## 何时跑

- 接手陌生中大型库，先做一次整体体检再动手。
- 重构/还债前，需要按「危险×价值」给候选排优先级。
- 定期债务巡检（复跑上次报告，标 RESOLVED / NEW，做成活文档）。

## 核心机制：churn × complexity 找热点

技术债不是「哪段代码丑」，是**「改得最勤 × 最难改」的交集**——那里每次改动都贵、且最可能出 bug。单看复杂度会冤枉「复杂但稳定不动」的代码（如成熟的解析器）；单看改动频率会冤枉「常改但简单」的代码（如配置）。**取交集**才是真热点。

四步流水（命令与各栈工具见 `references/hotspot-mining.md`）：

1. **挖 churn**：`git log` 统计近 6–12 个月各文件改动次数，排序。
2. **量 complexity**：god-file（行数）、超长函数、循环依赖、重复块、无测试覆盖区——按栈派对应扫描器。
3. **求交集打分**：churn × complexity 排序，高分进热点榜；有覆盖率数据则纳入（churn×complexity÷coverage，越无测试越危险）。
4. **逐条钉证据**：每个热点深读，落到 `file:line`，写清影响 + 严重度 + 建议。

**大库（>50k LOC）防上下文爆**：用 `Task` 工具按目录/模块派子代理分头审，每个子代理只回传**结构化发现条目**（file:line + 影响 + 严重度），主线程汇总打分、不吞全部源码。子代理回传格式见 references。

## 反幻觉铁律（不可违反）

1. **每条发现强制带 `file:line` 证据。** 拒绝「the code generally…」「这块整体偏乱」「耦合有点高」这类无定位空话——没有具体行号坐标的发现，一律删除或降级为「待核实」，不进报告主表。看着像问题 ≠ 是问题。
2. **必含一段「看着差其实合理」分析。** 报告里强制留一节，列出**你考虑过但决定不flag**的代码及理由——防止浅层模式匹配把正常代码（防御性冗余、刻意的性能 hack、稳定的复杂解析器）误判成债。漏了这节 = 审计不合格。
3. **严重度按 churn×complexity 客观分级，不凭手感。** 高 churn × 高 complexity = 高危；低 churn 的复杂代码即便丑也降级（它不常改，痛感低）。分级理由要能从信号推出来。
4. **审计只诊断、不开药、不改代码。** 给位置 + 影响 + 建议方向；具体怎么重构归 `plan` → `implement`。在报告里写完整重构方案 = 越界。
5. **不查外网。** 债务证据全部来自本仓库（git 历史 + 源码 + 测试）。需要外部技术调研是 `research` 的活。

## 「看着差其实合理」红线（防误报）

浅层模式匹配会把一堆正常代码当债。下表是 agent 原话级的合理化借口 → 实际规则：

| 合理化借口（agent 会这么想） | 实际规则 |
|---|---|
| 这文件 1200 行，god-file，标高危 | 行数只是信号不是判决。先查 churn：半年没人动的 1200 行解析器是「稳定」不是「债」。无 churn 不进高危。 |
| 这段没单测，覆盖率 0%，必须 flag | 先问它**变不变、复杂不复杂**。零覆盖 + 零 churn + 低复杂 = 低优先；零覆盖 + 高 churn + 高复杂才是真窟窿。 |
| 这里重复了三次，DRY，建议抽象 | 重复有时比错误抽象便宜。确认三处是否真同构、是否会一起变；偶然雷同的代码强行抽象反而造耦合债。标注，别拍板。 |
| 这函数嵌套很深，圈复杂度高，标债 | 查它改动频率与 bug 历史。深但稳定、且有测试兜底的，痛感低，降级。复杂≠债，复杂×常改才是。 |
| 整个 service 层看着乱，整体标「需重构」 | 「整体乱」是空话（违铁律1）。逐文件给 file:line，给不出坐标的就不是发现，是印象。 |
| 这块用了反常写法，肯定是债 | 可能是刻意的性能/兼容 hack。读 commit message 与注释找意图，没确认意图前归入「看着差其实合理」待核，不进高危。 |
| 既然发现了问题，顺手把重构方案也写全 | 越界（违铁律4）。审计给位置+方向，方案归 plan。 |
| 复跑时上次的条目还在，直接沿用旧报告 | 复跑必须重挖 churn、重核每条：修好的标 RESOLVED，新增的标 NEW，没动的复核仍成立。沿用 = 假活文档。 |

## 危险信号（出现即停 / 回退）

- 报告里出现没有 `file:line` 的「发现」→ 停，要么补坐标要么删。
- 你正把一个**半年没改动**的复杂文件标成高危 → 回退，churn 不支持高危。
- 报告缺「看着差其实合理」一节 → 不合格，补上再交。
- 你开始在报告里写详细重构代码/方案 → 越界，收回，交给 plan。
- 大库你试图把所有源码读进主上下文 → 停，派子代理分模块。
- 复跑时你直接复制上次条目没重新核验 → 停，逐条标 RESOLVED/NEW/仍存在。

## 正例 / 反例

**反例（浅层、空话、误报）：**
> 「`src/payment/` 整体复杂度偏高，建议重构。`OrderService` 这个类太大了，god-class。很多地方没有测试覆盖，技术债较重。」
—— 无 file:line（违铁律1）、把「大」当「债」没看 churn（违红线）、无「看着差其实合理」节、给不出优先级。

**正例（钉证据、按信号分级、留反节）：**
> **高危** `src/payment/refund.go:88-241`（churn: 近半年 31 次提交 / 全库 top3，圈复杂度 42，覆盖率 0%）：`processRefund` 单函数串了校验+扣款+对账+通知四职责，每次改退款逻辑都动它且无测试兜底——这是「改得最勤×最难改×无网」的三重热点，建议拆职责并补集成测试（具体拆法交 plan）。
> **看着差其实合理**：`src/parser/grammar.go:1-900`（900 行，圈复杂度高）churn=0（11 个月未改）、有完整测试——稳定的语法解析器，复杂是本质复杂度，不是债，**不flag**。

## 产物：根目录 `TECH_DEBT_AUDIT.md`（可复跑活文档）

可提交、可复跑。复跑时不新建文件，在原文档上更新状态。模板：

```markdown
# 技术债审计报告
> 由 tech-debt-audit 维护。可复跑：复跑时标 RESOLVED / NEW，逐条重核。
- 审计日期: <date>   提交: <git short sha>   规模: <LOC>
- 信号源: git churn(近 N 月) + <复杂度工具> + <覆盖率，若有>

## 风险表（按 churn×complexity 降序）
| ID | 状态 | 位置 file:line | 类别 | churn | 复杂度 | 覆盖率 | 影响 | 严重度 | 建议方向 |
|----|------|----------------|------|-------|--------|--------|------|--------|----------|
| D1 | 仍存在 | src/payment/refund.go:88 | god-fn/无测试 | 31 | 42 | 0% | 退款逻辑高频改且无网 | 高危 | 拆职责+补集成测试 |
| D2 | RESOLVED(2026-..) | ~~src/auth/token.go:..~~ | 循环依赖 | - | - | - | 已解 | - | - |
| D3 | NEW | src/sync/queue.go:..  | 重复块×3 | 12 | 18 | 40% | … | 中 | … |

## 热点榜（churn × complexity top N）
<排序后的文件清单 + 分数>

## 看着差其实合理（考虑过但不flag，含理由）  ← 强制节，不可省
- src/parser/grammar.go:1-900 — 复杂但 churn=0 且有测试，本质复杂度，不是债。
- …

## 复跑摘要（第二次起填）
- RESOLVED: <上次有、这次已解>
- NEW: <这次新增>
- 仍存在: <持续未还>
```

严重度四档：高危 / 中 / 低 / 观察。分级必须由 churn×complexity(÷coverage) 推出，不凭手感（铁律3）。条目数控制在信噪比内（约 20–80 条），别堆噪音。

## 完成判定

审计不走代码型 `verify`/Oracle（不改代码无可执行断言）。完成的标准是**报告自洽**：①每条主表发现都有 `file:line`；②严重度都能从信号推出；③「看着差其实合理」节非空；④若为复跑，每条都已标 RESOLVED/NEW/仍存在。缺一不算完成。
